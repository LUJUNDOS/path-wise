## 二、Prisma Migrate 使用规范

### 2.1 开发环境迁移

#### 开发流程

```bash
# 1. 修改 schema.prisma
# 编辑 apps/api/prisma/schema.prisma

# 2. 生成迁移文件
pnpm --filter api prisma migrate dev --name add_user_table

# 这会自动：
# - 在 prisma/migrations/ 创建迁移文件
# - 更新数据库 Schema
# - 更新 @prisma/client

# 3. 验证迁移
pnpm --filter api prisma studio

# 4. 提交迁移文件
git add prisma/migrations/
git commit -m "feat: add user table"
```

#### 迁移文件结构

```
prisma/migrations/
├── 20250621030000_add_user_table/
│   ├── migration.sql       # SQL 迁移脚本
│   └── README.md         # 可选：迁移说明
├── 20250622080000_add_trip_table/
│   └── migration.sql
└── migration_lock.toml   # 迁移锁文件（不要手动修改）
```

### 2.2 生产环境迁移

#### 生产迁移流程

```bash
# 1. 备份生产数据库
pg_dump $DATABASE_URL > backup_$(date +%Y%m%d_%H%M%S).sql

# 2. 应用迁移（dry-run 模式，只生成 SQL，不执行）
pnpm --filter api prisma migrate deploy --dry-run

# 3. 检查生成的 SQL，确认无误后执行
pnpm --filter api prisma migrate deploy

# 4. 验证迁移
pnpm --filter api prisma studio
```

#### CI/CD 自动化迁移

```yaml
# .github/workflows/deploy.yml

name: Deploy to Production

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3

      - name: Setup Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '22'

      - name: Install dependencies
        run: pnpm install

      - name: Backup database
        run: |
          pg_dump $DATABASE_URL > backup_$(date +%Y%m%d_%H%M%S).sql
          # 上传备份到云存储（如 S3）

      - name: Apply database migrations
        run: pnpm --filter api prisma migrate deploy

      - name: Deploy application
        run: pnpm --filter api deploy
```


## 四、Schema 变更管理

### 4.1 变更分类

| 变更类型         | 风险等级  | 迁移策略                  | 回滚策略                  |
| ---------------- | --------- | ------------------------- | ------------------------- |
| **新增表/字段**  | 🟢 低风险 | 直接迁移                  | 删除表/字段（会丢失数据） |
| **修改字段类型** | 🟡 中风险 | 需要数据迁移              | 复杂（可能需要双写）      |
| **删除字段**     | 🔴 高风险 | 先标记为废弃，再删除      | 从备份恢复                |
| **修改表名**     | 🔴 高风险 | 使用 `ALTER TABLE RENAME` | 同上                      |

### 4.2 安全变更流程（中高风险）

```
Step 1：在开发环境测试迁移
  │
  ▼
Step 2：生成迁移 SQL，人工审查
  │
  ▼
Step 3：在 Staging 环境执行迁移
  │
  ▼
Step 4：验证 Staging 环境功能
  │
  ▼
Step 5：备份生产数据库
  │
  ▼
Step 6：在生产环境执行迁移（业务低峰期）
  │
  ▼
Step 7：验证生产环境功能
  │
  ▼
Step 8：监控错误日志和性能指标
```

## 五、数据备份与恢复

### 5.1 备份策略

| 备份类型     | 频率       | 保留时间 | 存储位置            |
| ------------ | ---------- | -------- | ------------------- |
| **全量备份** | 每日 02:00 | 30 天    | 本地 + 云存储（S3） |
| **增量备份** | 每 6 小时  | 7 天     | 本地                |
| **WAL 归档** | 实时       | 7 天     | 云存储（S3）        |

#### 自动化备份脚本

```bash
# scripts/backup-database.sh

#!/bin/bash

# 配置
DATABASE_URL=$DATABASE_URL
BACKUP_DIR="/var/backups/path-wise"
S3_BUCKET="s3://path-wise-backups"
DATE=$(date +%Y%m%d_%H%M%S)
BACKUP_FILE="$BACKUP_DIR/backup_$DATE.sql"

# 创建备份目录
mkdir -p $BACKUP_DIR

# 执行备份
echo "开始备份数据库..."
pg_dump $DATABASE_URL > $BACKUP_FILE

if [ $? -eq 0 ]; then
  echo "✅ 备份成功: $BACKUP_FILE"

  # 压缩备份文件
  gzip $BACKUP_FILE

  # 上传到 S3
  aws s3 cp $BACKUP_FILE.gz $S3_BUCKET/

  if [ $? -eq 0 ]; then
    echo "✅ 备份已上传到 S3"
  else
    echo "❌ 上传到 S3 失败"
  fi
else
  echo "❌ 备份失败"
  exit 1
fi

# 清理旧备份（保留 30 天）
find $BACKUP_DIR -name "backup_*.sql.gz" -mtime +30 -delete
echo "✅ 旧备份清理完成"
```

#### 定时任务配置

```bash
# 添加到 crontab（每日 02:00 执行）
0 2 * * * /path/to/scripts/backup-database.sh >> /var/log/path-wise-backup.log 2>&1
```

### 5.2 恢复策略

#### 恢复流程

```bash
# 1. 停止应用（避免数据不一致）
pm2 stop path-wise-api

# 2. 恢复数据库
psql $DATABASE_URL < backup_20250621_030000.sql

# 3. 验证数据完整性
pnpm --filter api prisma studio

# 4. 重启应用
pm2 start path-wise-api
```

#### 时间点恢复（PITR）

```bash
# 1. 恢复最近的全量备份
psql $DATABASE_URL < backup_20250620_020000.sql

# 2. 应用 WAL 日志到指定时间点
pg_rewind --target-time="2026-06-21 03:00:00" ...
```

## 六、种子数据管理

### 6.1 种子数据分类

| 数据类型       | 用途       | 更新频率 | 管理方式     |
| -------------- | ---------- | -------- | ------------ |
| **城市知识库** | 静态数据   | 季度更新 | 独立迁移文件 |
| **管理员账号** | 系统初始化 | 仅初始化 | 独立迁移文件 |
| **配置数据**   | 系统配置   | 按需更新 | 独立迁移文件 |

### 6.2 种子数据脚本

```typescript
// prisma/seed.ts

import { PrismaClient } from '@prisma/client';
import * as fs from 'fs';
import * as path from 'path';

const prisma = new PrismaClient();

async function main() {
  // 读取城市知识库数据
  const cityData = JSON.parse(
    fs.readFileSync(path.join(__dirname, 'seed-data/cities.json'), 'utf-8'),
  );

  // 插入城市知识库数据
  for (const city of cityData) {
    await prisma.cityKnowledge.upsert({
      where: { cityName: city.cityName },
      update: city,
      create: city,
    });
  }

  console.log(`✅ 种子数据插入完成: ${cityData.length} 个城市`);
}

main()
  .catch((e) => {
    console.error('种子数据插入失败:', e);
    process.exit(1);
  })
  .finally(async () => {
    await prisma.$disconnect();
  });
```

#### 执行种子数据

```bash
# 执行种子数据
pnpm --filter api prisma db seed

# 配置 package.json
{
  "prisma": {
    "seed": "tsx prisma/seed.ts"
  }
}
```

## 九、回滚策略

### 9.1 回滚场景

| 场景                     | 回滚方式            | 数据丢失风险               |
| ------------------------ | ------------------- | -------------------------- |
| **迁移失败（未应用）**   | 无需回滚            | 无                         |
| **迁移成功，但应用出错** | 回滚迁移 + 回滚代码 | 有（如果迁移包含删除操作） |
| **数据损坏**             | 从备份恢复          | 恢复到备份时间点           |
| **误删除数据**           | 从备份恢复          | 恢复到备份时间点           |

### 9.2 回滚迁移

#### 方法 1：使用 `prisma migrate resolve`（推荐）

```bash
# 1. 标记迁移为已回滚（不改数据库）
pnpm --filter api prisma migrate resolve --rolled-back 20250621030000_add_user_table

# 2. 创建新的迁移（撤销之前的变更）
pnpm --filter api prisma migrate dev --name rollback_add_user_table

# 这会生成反向迁移 SQL
```

#### 方法 2：手动编写反向迁移

```sql
-- prisma/migrations/20250621040000_rollback_add_user_table/migration.sql

-- 手动编写反向迁移
DROP TABLE IF EXISTS "User";
```

## 十、性能优化

### 10.1 索引策略

```prisma
// prisma/schema.prisma

model Trip {
  id        Int      @id @default(autoincrement())
  userId    Int?
  title     String
  cityFrom  String
  cityTo    String[]
  startDate DateTime
  endDate   DateTime
  status    String

  user User? @relation(fields: [userId], references: [id])

  @@index([userId])               // 外键索引
  @@index([cityTo], type: Gist)  // 数组字段索引（PostgreSQL GiST）
  @@index([startDate, endDate])  // 复合索引
  @@index([status])               // 状态索引（查询进行中的行程）
}
```

### 10.2 查询优化

```typescript
// 使用 select 减少返回字段
const trips = await prisma.trip.findMany({
  where: { userId: 1 },
  select: {
    id: true,
    title: true,
    startDate: true,
    endDate: true,
    status: true,
    // 不返回 dayPlans（JSONB 字段，可能很大）
  },
});

// 使用 include 预加载关联数据
const trip = await prisma.trip.findUnique({
  where: { id: 1 },
  include: {
    user: true, // 预加载用户数据
  },
});
```

## 十一、监控与告警

### 11.1 关键指标

| 指标             | 说明                 | 告警阈值         |
| ---------------- | -------------------- | ---------------- |
| **数据库连接数** | 当前活跃连接数       | > 80% 最大连接数 |
| **慢查询数量**   | 执行时间 > 1s 的查询 | > 10/分钟        |
| **迁移失败次数** | 迁移失败计数         | > 0              |
| **数据库大小**   | 数据库文件大小       | > 80% 磁盘容量   |

### 11.2 PostgreSQL 监控配置

```sql
-- 启用 pg_stat_statements（慢查询监控）
CREATE EXTENSION pg_stat_statements;

-- 查询慢查询
SELECT query, mean_time, calls
FROM pg_stat_statements
ORDER BY mean_time DESC
LIMIT 10;
```
