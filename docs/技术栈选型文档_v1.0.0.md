## 四、LLM 智能路由策略

### 4.1 模型能力矩阵

```typescript
const LLM_CAPABILITY_MATRIX = {
  'deepseek-v2': {
    strengths: ['code_generation', 'logical_reasoning', 'cost_efficiency'],
    contextWindow: 128000,
    costPer1KTokens: 0.14, // 人民币
    speed: 'fast',
    bestFor: ['itinerary_generation', 'logical_optimization'],
  },
  'glm-4': {
    strengths: ['chinese_comprehension', 'cultural_context', 'long_text'],
    contextWindow: 128000,
    costPer1KTokens: 0.1,
    speed: 'balanced',
    bestFor: ['poi_recommendation', 'local_culture'],
  },
  kimi: {
    strengths: ['long_context', 'document_analysis', 'detail_extraction'],
    contextWindow: 200000,
    costPer1KTokens: 0.12,
    speed: 'medium',
    bestFor: ['complex_itinerary', 'multi_day_planning'],
  },
  mimo: {
    strengths: ['lightweight', 'fast_response', 'simple_tasks'],
    contextWindow: 32000,
    costPer1KTokens: 0.05,
    speed: 'very_fast',
    bestFor: ['quick_suggestion', 'simple_qa'],
  },
};
```

### 4.2 路由逻辑

```typescript
function routeLLM(config: LLMRouterConfig): string {
  const { taskType, inputTokens, costPriority, speedPriority } = config;

  // 规则 1：根据任务类型
  if (taskType === 'itinerary_generation') {
    if (inputTokens > 100000) return 'kimi'; // 长上下文
    return 'deepseek-v2'; // 逻辑能力强
  }

  if (taskType === 'poi_recommendation') {
    return 'glm-4'; // 中文理解好
  }

  // 规则 2：根据成本优先级
  if (costPriority === 'low') return 'mimo';

  // 规则 3：根据速度优先级
  if (speedPriority === 'fast') return 'deepseek-v2';

  // 默认：DeepSeek（性价比最高）
  return 'deepseek-v2';
}
```

## 八、技术栈决策记录

### 8.1 决策 1：前端 UI 框架选择 shadcn/ui

**背景**：需要现代化、可定制的 UI 组件库。

**选项**：

- A：Ant Design 5.x（企业级）
- B：shadcn/ui + Tailwind CSS（现代化）
- C：MUI（Material Design）

**决策**：选择 B（shadcn/ui）

**理由**：

1. 完全可定制（代码复制，不是 npm 包）
2. 现代化设计（符合 2026 审美）
3. 无障碍支持（WCAG 2.1 AA）
4. 与 Tailwind CSS 原生集成

### 8.2 决策 2：后端框架选择 Fastify

**背景**：需要高性能、支持 SSE 的 Web 框架。

**选项**：

- A：Express（成熟但性能一般）
- B：Fastify（高性能，原生 SSE 支持）
- C：Nest.js（企业级，但学习曲线陡）

**决策**：选择 B（Fastify）

**理由**：

1. 性能优秀（比 Express 快 2x）
2. 原生支持 SSE（无需额外插件）
3. 原生支持 JSON Schema 验证
4. 插件化架构（易于扩展）

### 8.3 决策 3：LLM 多模型智能路由

**背景**：单一 LLM 无法兼顾成本、速度、质量。

**选项**：

- A：只使用 GPT-4（质量高但成本高）
- B：只使用 DeepSeek（成本低但中文理解一般）
- C：多模型智能路由（根据任务动态选择）

**决策**：选择 C（多模型智能路由）

**理由**：

1. 成本优化（简单任务用 MiMo，复杂任务用 DeepSeek）
2. 速度优化（快速响应用 DeepSeek，长文本用 Kimi）
3. 质量优化（中文 POI 推荐用 GLM-4）

## 九、风险评估

### 9.1 技术风险

| 风险                       | 影响                       | 应对措施                           |
| -------------------------- | -------------------------- | ---------------------------------- |
| **12306 非官方接口不稳定** | MVP 阶段无法获取火车票数据 | 阶段一切换到静态数据（城市知识库） |
| **LLM API 成本失控**       | 运营成本超预算             | 实施成本监控，设置每日上限         |
| **高德 API 额度耗尽**      | POI 搜索功能不可用         | 实施缓存策略，减少重复请求         |
| **Windows 开发环境兼容性** | Docker/Git 权限问题        | 使用 WSL2 或管理员权限运行         |

### 9.2 业务风险

| 风险                        | 影响             | 应对措施                        |
| --------------------------- | ---------------- | ------------------------------- |
| **OTA 平台 API 不开放个人** | 无法实现在线订票 | MVP 阶段只做推荐，跳转 OTA 平台 |
| **用户留存率低**            | 产品无法商业化   | 阶段二增加社交分享、UGC 内容    |
