## 1. 概述

### 1.1 文档目的

本文档定义旅游攻略生成平台 MVP 阶段的**前端交互设计规范**，作为前端开发的直接依据。文档描述"用户看到什么、能做什么、系统如何响应"，不包含具体实现代码。

### 1.2 适用范围

- MVP 阶段（单城市 → 多城市渐进）
- Web 端（桌面 + 平板 + 手机自适应）
- 不涉及小程序/App 端（阶段二扩展）

### 1.3 读者对象

| 角色           | 使用方式                   |
| -------------- | -------------------------- |
| 前端开发工程师 | 依据本文档实现组件与交互   |
| UI 设计师      | 依据本文档输出高保真设计稿 |
| 产品经理       | 依据本文档验收交互效果     |
| 测试工程师     | 依据本文档编写测试用例     |

### 1.4 引用文档

| 文档名称                       | 版本   | 关键引用内容                     |
| ------------------------------ | ------ | -------------------------------- |
| 旅游攻略生成平台\_SRS.md       | v1.1.0 | 功能需求、用户角色、业务流程     |
| API接口设计规格书\_v1.0.0.md   | v1.0.5 | SSE 事件格式、接口协议、数据结构 |
| Trip*Lifecycle*引擎算法设计.md | v1.0.0 | 日类型分类、行程生成逻辑         |
| 数据库设计规格书\_v1.0.0.md    | v1.0.3 | 数据模型（前端状态管理参考）     |

### 1.5 MVP 阶段关键约束

| 约束                     | 说明                                                        |
| ------------------------ | ----------------------------------------------------------- |
| **无 LLM 缓存**          | MVP 阶段已去掉缓存，每次生成都是全新请求                    |
| **SSE 流式返回**         | 生成耗时 30~120s，必须流式渲染，不可用传统 HTTP 轮询        |
| **冲突提示非阻断**       | 6 条冲突规则均为"提示+可忽略"，不强制拦截用户选择           |
| **大交通为静态数据**     | MVP 阶段不调实时 API，展示静态经验推荐并标注免责声明        |
| **模型选择对用户不可见** | 平台统一提供 API Key，前端仅可选展示"当前使用：DeepSeek-V3" |

### 1.6 术语与缩写（阅读前必读）

在阅读本文档前，请先阅读本节，了解常用术语与缩写。

| 术语 / 缩写       | 英文全称               | 说明                                                     |
| ----------------- | ---------------------- | -------------------------------------------------------- |
| **MVP**           | Minimum Viable Product | 最小可行产品，本文档中"MVP 阶段"指第一版上线功能范围     |
| **SSE**           | Server-Sent Events     | 服务端推送事件，用于流式返回攻略生成过程                 |
| **POI**           | Point of Interest      | 兴趣点，指景点、餐厅、酒店等地点                         |
| **Day Card**      | -                      | 天级卡片，前端展示单天行程的容器组件                     |
| **Timeline Item** | -                      | 时间轴条目，卡片内的单个活动项（如"09:00-12:00 岳麓山"） |
| **accommodation** | -                      | 住宿（拼写较长，文中缩写为 `accommodation`）             |
| **shareToken**    | -                      | 分享令牌，用于生成分享链接时的权限标识                   |
| **僵死连接**      | -                      | SSE 连接建立后，服务端长时间不返回任何事件的异常状态     |
| **渐进渲染**      | -                      | 收到 `day_ready` 事件后立即渲染当天行程，不用等全部完成  |
| **冲突检测**      | -                      | 用户提交生成请求时，前端预检 + 后端校验可能的不合理组合  |
| **非阻断提示**    | -                      | 冲突检测结果仅为提示，不强制拦截用户选择                 |

### 3.1 首页（城市选择 + 日期 + 人数 + 快速偏好）

#### 3.1.1 页面元素清单

| 元素 ID | 元素名称         | 类型                           | 必填 | 说明                                                           |
| ------- | ---------------- | ------------------------------ | ---- | -------------------------------------------------------------- |
| E-001   | 出发城市输入框   | 搜索选择框                     | ✅   | 支持城市名/拼音/缩写模糊搜索                                   |
| E-002   | 目的地城市输入框 | 搜索选择框（可多个）+ 停留天数 | ✅   | 支持添加多个（≤5个），每个带停留天数选择器（1~14天）和删除按钮 |
| E-003   | 出发日期选择器   | 日期选择                       | ✅   | 默认今天，不可选过去日期                                       |
| E-004   | 出发时间段选择   | 单选（早/中/晚）               | ✅   | 默认"上午"                                                     |
| E-005   | 成人人数计数器   | 数字步进器                     | ✅   | 默认 1，范围 1~10                                              |
| E-006   | 儿童人数计数器   | 数字步进器                     | ❌   | 默认 0，点击后展开年龄选择                                     |
| E-007   | 老人人数计数器   | 数字步进器                     | ❌   | 默认 0                                                         |
| E-008   | 快速偏好展开按钮 | 按钮                           | ❌   | 点击展开完整偏好设置（内联展开，不跳转）                       |
| E-009   | 生成攻略按钮     | 主按钮                         | -    | 大号，绿色，全宽（移动端）/ 固定宽度（桌面端）                 |
| E-010   | 历史攻略入口     | 链接/卡片                      | ❌   | 展示最近 3 条历史攻略，点击直接进入结果页                      |

#### 3.1.2 交互流程

**主流程**：

```
1. 用户进入首页，焦点自动定位到"出发城市"输入框
2. 用户输入出发城市（输入时实时搜索，显示下拉候选列表）
3. 用户选择目的地城市（可添加多个，每个目的地带"×"删除按钮）
4. 用户设置出发日期和时间段
5. 用户设置出行人数（成人必填，儿童/老人选填）
6. 用户可选点击"更多偏好"展开完整偏好设置（内联展开，不跳转页面）
7. 用户点击"生成攻略"按钮
8. 前端执行输入校验（见 3.1.3）
9. 校验通过后，前端调用 POST /trips/validate 接口检测冲突
10. 无冲突 → 跳转生成中页面
11. 有冲突 → 弹出冲突提示弹窗（见 3.3 节）
```

**目的地城市交互细节**：

- 添加目的地：点击"添加目的地"按钮 → 出现新的城市选择框 + 停留天数选择器
- 停留天数：每个目的地 tag 内嵌天数步进器（默认 2 天，范围 1~14 天）
  - 1 天 = 途经/打卡（引擎自动安排少量核心 POI）
  - 2~3 天 = 常规游玩（推荐档位）
  - 4+ 天 = 深度游玩（引擎安排更多小众 POI 和周边探索）
- 目的地排序：支持拖拽排序（桌面端）；移动端通过"上移/下移"按钮调整
- 最多支持 5 个目的地（MVP 阶段限制，防止生成时间过长）
- 总天数实时汇总：底部显示「共 X 天」（含引擎自动插入的中转日）

**返程交通交互细节**：

- 默认勾选"预定返程"（checkbox），用户可取消
  - 勾选时：显示返程交通类型下拉框，选项：智能推荐 / 高铁 / 普速火车 / 航班 / 大巴
  - 取消时：隐藏返程交通类型下拉框，攻略生成时不注入返程大交通信息
- 返程自动计算：末站城市 → 出发城市的返程大交通
- 返程日期由引擎自动推算（出发日 + 总停留天数），用户无需手选
- 未勾选时，最后一日作为普通游玩日展示；生成的大交通信息卡仅显示去程

#### 3.1.3 校验规则

| 校验项             | 规则           | 提示方式                                                       |
| ------------------ | -------------- | -------------------------------------------------------------- |
| 出发城市           | 不能为空       | 输入框红框 + Toast："请选择出发城市"                           |
| 目的地城市         | 至少 1 个      | Toast："请至少选择 1 个目的地"                                 |
| 出发日期           | 不能选过去日期 | 日期选择器禁用过去日期                                         |
| 成人人数           | ≥ 1            | 步进器最小值 1，不可调至 0                                     |
| 目的地数量         | ≤ 5            | 添加第 6 个时 Toast："MVP 阶段最多支持 5 个目的地"             |
| 每个目的地停留天数 | 1~14 天        | 天数 < 1 时自动设为 1；> 14 时 Toast："单个城市最多停留 14 天" |
| 总天数             | ≤ 30 天        | 总天数 > 30 时 Toast："MVP 阶段最多支持 30 天行程"             |

### 3.3 冲突检测提示交互

#### 3.3.1 触发时机

| 触发点                            | 说明                                             |
| --------------------------------- | ------------------------------------------------ |
| **点击"生成攻略"按钮后**          | 前端先调用 `POST /trips/validate` 接口，检测冲突 |
| **接口返回 `conflicts` 数组非空** | 弹出冲突提示弹窗，不自动跳转生成中页面           |
| **接口返回 `conflicts` 数组为空** | 不弹窗，直接跳转生成中页面                       |

> **重要**：冲突检测是"提示"而非"阻断"，用户始终可以选择"坚持生成"。

**组件类型**：模态框（Modal），覆盖在首页或偏好设置页上方

**视觉结构**：

```

│  │    [保持当前选择]  [调整为经济型]   │  │

│  │ 2. 同行有老人，高强度节奏可能       │  │
```

**交互规则**：

| 用户操作                      | 系统响应                                     |
| ----------------------------- | -------------------------------------------- |
| 点击单条冲突的"调整为XXX"按钮 | 自动调整对应字段，该条冲突从列表中消失       |
| 点击"保持当前选择"            | 该条冲突标记为"用户已知悉"，变灰显示         |
| 点击"坚持生成"                | 关闭弹窗，携带 `conflicts` 字段发起生成请求  |

#### 3.3.3 6 条规则的交互细则

| 规则 # | 冲突条件                                       | 提示方式     | 可选操作              |
| ------ | ---------------------------------------------- | ------------ | --------------------- |
| 1      | `budget=穷游` + `accommodation=精品/豪华`      | Toast + 弹窗 | 保持 / 调整为经济型   |
| 2      | `budget=穷游` + `dining` 含"高端餐厅"          | Toast + 弹窗 | 保持 / 去掉高端餐厅   |
| 3      | `travelers.elders` 非空 + `pace=高强度`        | 弹窗（重要） | 保持 / 调整为舒适     |
| 4      | `children` 有 `<3岁` + `transportTo` 含高铁>4h | 弹窗（重要） | 保持 / 查看航班替代   |
| 5      | `pace=高强度` + `travelers` 有老幼             | 弹窗（重要） | 保持 / 调整为舒适     |
| 6      | `budget=豪华` + `accommodation=青旅/民宿`      | Toast 提示   | 保持 / 调整为精品酒店 |

**前端判断逻辑（伪代码）**：

```javascript
// 注意：此为交互逻辑描述，非实现代码
// 前端在调用 /trips/validate 之前，可做本地预检，快速提示常见冲突

function detectConflictsLocally(formData) {
  const conflicts = [];

  // 规则 1
  if (
    formData.budget === "economy" &&
    ["boutique", "luxury"].includes(formData.accommodation)
  ) {
    conflicts.push({
      type: "budget_accommodation",
      severity: "warning",
      message: "穷游预算下精品/豪华住宿可能超支，建议调整",
    });
  }

  // 规则 3
  if (formData.elders.length > 0 && formData.pace === "intensive") {
    conflicts.push({
      type: "pace_elders",
      severity: "warning",
      message: "同行有老人，高强度节奏可能不适合，建议调整为舒适或悠闲",
    });
  }

  // ... 其他规则

  return conflicts;
}
```

> **说明**：本地预检用于提升响应速度，但最终以服务端 `/trips/validate` 接口返回为准（服务端可能有更复杂的冲突检测逻辑）。

### 3.4 生成中页面（SSE 进度条 + 渐进渲染）

> **核心设计目标**：解决"用户等待焦虑"问题。通过 SSE 流式返回，让用户看到"每天都在逐步生成"，而非干等。

#### 3.4.1 页面状态机

```
        ┌──────────┐
        │   idle   │  初始状态（刚进入页面）
        └────┬─────┘
             │ 发起 SSE 连接
             ▼
        ┌──────────┐
        │connecting│  正在建立 SSE 连接（显示"正在连接..."）
        └────┬─────┘
             │ 收到 connected 事件
             ▼
        ┌──────────┐
        │streaming │  生成中，接收 progress 和 day_ready 事件
        └────┬─────┘
             │ 收到 done 事件
             ▼
        ┌──────────┐
        │all_complete│  (全部完成)
        └────┬──────┘
             │
             ▼
        ┌──────────┐
        │ redirect  │  跳转到行程结果页
        └──────────┘

  (任何时候，收到 error 事件或超时 → 转移到 error 状态)
```

**状态说明**：

- `idle`：初始状态
- `connecting`：正在建立 SSE 连接
- `streaming`：生成进行中，接收 `progress`、`day_ready`、`warning`、`error` 事件
  - 收到 `day_ready` 事件：渲染该天卡片（渐进渲染），**状态保持 `streaming` 不变**
  - 收到 `done` 事件：转移到 `all_complete`
- `all_complete`：全部生成完成，准备跳转
- `redirect`：跳转到行程结果页
- `error`：生成失败（收到 `error` 事件或超时）

#### 3.4.2 进度条规范

**进度条类型**：分步进度条（显示当前步骤 / 总步骤）

**视觉设计**：

```
┌──────────────────────────────────────────────────────┐

│                                                      │
│  Step 3 / 12   正在为 Day 2 安排景点...               │
│  ████████░░░░░░░░░░░░░░░  25%                        │
│                                                      │
│  💡 小提示：长沙的茶颜悦色值得一试！                   │
│                                                      │
│  ─────────────────────────────────────────────────   │
│  已生成：                                             │
│  ✅ Day 1 · 抵达长沙                                  │
│  ✅ Day 2 · 长沙深度游   ← 刚完成                     │
```

**进度计算规则**：

| 字段            | 来源                 | 说明                                               |
| --------------- | -------------------- | -------------------------------------------------- |
| `step`          | SSE `progress` 事件  | 当前步骤编号                                       |

**进度条动画**：

- 进度条填充动画：300ms 缓动

#### 3.4.4 渐进渲染区域

**核心机制**：收到 `day_ready` 事件后，立即渲染该天卡片，不用等全部完成。

**渲染规则**：

| 时机                  | 操作                                                     |
| --------------------- | -------------------------------------------------------- |
| 收到 `connected` 事件 | 初始化空白的天级卡片占位区（显示"等待生成..."）          |
| 收到 `day_ready` 事件 | 将该天卡片从"等待中"状态更新为"已完成"状态，填入行程数据 |
| 收到 `done` 事件      | 所有天卡片已完成，显示"生成完成"按钮，3 秒后自动跳转     |

**天级卡片占位区设计（生成中状态）**：

```
┌──────────────────────────────────────────────────────┐
```

### 3.8 分享与导出交互

#### 3.8.1 分享功能

**触发时机**：

- 行程结果页点击"分享"按钮
- 生成完成后（`done` 事件返回 `shareUrl`）

**分享渠道**（MVP 阶段）：

| 渠道       | 方式                                  | 说明                           |
| ---------- | ------------------------------------- | ------------------------------ |
| 微信好友   | 调用微信 JS-SDK（需在微信内浏览器中） | 分享卡片：标题 + 封面图 + 摘要 |
| 微信朋友圈 | 调用微信 JS-SDK                       | 分享图片（自动生成攻略长图）   |
| 复制链接   | 复制 `shareUrl` 到剪贴板              | `shareUrl` 来自 `done` 事件    |
| 生成图片   | 前端将攻略渲染为长图                  | 用户可保存图片分享到朋友圈     |

**分享交互流程**：

```
用户点击"分享"按钮
        ↓

  ┌──────────────────────────────────┐

  │                                  │

  │  [复制链接]    [生成图片]         │

用户选择"微信好友"
        ↓
1. 判断是否在微信内浏览器
2. 若在微信内：调用 wx.onMenuShareAppMessage
3. 若不在微信内：提示"请在微信中打开此链接后分享"
        ↓
分享成功：显示"分享成功"Toast
分享失败：显示"分享失败，请重试"Toast
```

### 3.9 微信小程序分享与协作修改机制

> **功能定位**：MVP 阶段的核心社交功能，允许多个用户协作完善一份攻略。
> **适用场景**：家庭出游、朋友组队、公司团建等需要多人协商行程的场景。

#### 3.9.1 核心流程概览

```
原用户：生成攻略
      ↓
点击"分享给微信好友"
      ↓
微信分享卡片（小程序卡片）
      ↓
同行人点击卡片 → 打开微信小程序 → 查看攻略
      ↓
同行人点击"建议修改" → 填写修改意见
      ↓
修改意见回传给原用户（服务通知 / 小程序内消息）
      ↓
原用户查看修改意见 → 选择"接受"或"忽略"
      ↓
接受 → 调用后端 API 更新攻略 → 重新生成受影响的天
```

#### 3.9.2 分享流程（原用户视角）

**触发时机**：行程结果页点击"分享"按钮，选择"微信好友"

#### 3.9.7 后端 API 设计（MVP 阶段）

| 接口名称                  | HTTP 方法 | 路径                                         | 权限要求          | 说明                                         |

| 生成分享 Token            | `POST`    | `/trips/{tripId}/share`                      | 仅原用户          | 生成 `shareToken`，设置权限和有效期          |
| 查看攻略（小程序）        | `GET`     | `/trips/{tripId}?shareToken=xxx`             | 持有 `shareToken` | 返回完整攻略 JSON，供小程序渲染              |

### 3.11 轻量智能引导（表单内智能推荐 + 生成前确认）

> **设计决策**：MVP 阶段不做完整对话式 UI（类 ChatGPT），而是在现有表单基础上增加 3 个「智能触点」，引导用户更高效地完成输入。对话式调优（生成后通过对话微调行程）推迟到阶段二。

#### 3.11.1 触点 1：智能推荐目的地

**触发时机**：用户添加一个目的地后（选中城市时）

**交互流程**：

**推荐数据来源**：后端 API （MVP 阶段可硬编码 TOP 20 热门城市的关系数据，阶段二接入真实地理距离计算）

**设计约束**：

- 推荐条不遮挡已选目的地列表
- 最多推荐 3 个城市
- 已添加的目的地不重复推荐
- 推荐条可手动关闭（右上角 × ）

#### 3.11.2 触点 2：智能天数建议

**触发时机**：用户为目的地选择停留天数时（点击天数步进器时）

**交互流程**：

**建议数据来源**：后端 API （MVP 阶段可硬编码热门城市推荐天数，阶段二基于 POI 数量动态计算）

**设计约束**：

- 气泡仅在天数步进器获得焦点时显示
- 气泡内容为静态文案（MVP 阶段不调用 LLM 生成，避免延迟）
- 气泡不遮挡其他输入框

#### 3.11.3 触点 3：生成前确认弹窗

**触发时机**：用户点击「生成攻略」按钮，校验通过后、调用 API 前

**弹窗内容**：

**弹窗元素**：

| 元素       | 类型   | 说明                                              |
| ---------- | ------ | ------------------------------------------------- |
| 路线摘要   | 文本   | 出发地 → 目的地1（X天）→ 目的地2（Y天）           |
| 出发信息   | 文本   | 日期 + 时间段                                     |
| 人数信息   | 文本   | 成人 X / 儿童 Y / 老人 Z                          |
| 预算信息   | 文本   | 档位名称 + 价格区间                               |
| 预计总天数 | 文本   | 各城市天数 + 中转日（引擎计算）                   |
| 预计总花费 | 文本   | 基于预算档位 × 天数 × 人数计算                    |
| 智能提示   | 列表   | 基于日期/目的地的静态提醒（旺季、天气、中转日等） |
| 返回调整   | 按钮   | 关闭弹窗，回到表单                                |
| 确认生成   | 主按钮 | 关闭弹窗，调用                                    |

**智能提示规则（MVP 阶段硬编码）**：

| 提示类型   | 触发条件                             | 提示内容示例                                    |
| ---------- | ------------------------------------ | ----------------------------------------------- |
| 中转日提示 | destinations.length ≥ 2              | 「Day X 为中转日（城市A→城市B），下午需赶高铁」 |
| 旺季提示   | 出发日期在 7-8 月或法定节假日        | 「X 月为旅游旺季，热门景点建议提前预约」        |
| 天气提示   | 目的地为北方城市且出发日期在 11-3 月 | 「冬季北方气温较低，注意保暖」                  |
| 长行程提示 | 总天数 > 10 天                       | 「行程较长，建议安排休息日」                    |

#### 3.11.4 设计约束与技术考量

| 约束项         | 说明                                                                                              |
| -------------- | ------------------------------------------------------------------------------------------------- |
| **不使用 LLM** | MVP 阶段所有智能引导均为规则驱动（硬编码 + 静态数据），不调用 LLM，确保零延迟                     |
| **可关闭**     | 所有智能引导元素均可手动关闭，不强制用户查看                                                      |
| **非阻塞**     | 智能引导不阻断用户操作流程，用户可以忽略所有引导直接生成                                          |
| **数据来源**   | 触点 1/2 的数据通过轻量 API 获取（返回 JSON，不涉及 LLM），触点 3 完全前端计算                    |
| **阶段二演进** | 阶段二可增加「攻略调优对话框」——生成完成后，用户通过对话微调行程（如「把 Day 2 的餐厅换成湘菜」） |

#### 3.11.5 涉及的 API 接口

| 接口         | 方法 | 说明                                     | 优先级             |
| ------------ | ---- | ---------------------------------------- | ------------------ |
|              | GET  | 获取附近热门城市（触点 1）               | P1（MVP 可硬编码） |
|              | GET  | 获取推荐天数 + 每日概览（触点 2）        | P1（MVP 可硬编码） |
| 前端本地计算 | -    | 生成前确认弹窗的路线/费用/提示（触点 3） | P0                 |

> **注意**：触点 1 和触点 2 的 API 为 P1 优先级，MVP 阶段可使用前端硬编码的静态数据（TOP 20 热门城市），不阻塞开发。触点 3 完全由前端计算，无额外 API 依赖。

## 4. SSE 流式渲染技术规范

> **本章是最关键章节**，详细描述前端如何解析 SSE 事件流并做渐进渲染。

### 4.1 事件类型定义

根据 `API接口设计规格书_v1.0.0.md` 第 5 章，SSE 事件流包含以下事件类型：

| 事件类型    | 触发时机                           | 前端必须处理  | 详细结构见 |
| ----------- | ---------------------------------- | ------------- | ---------- |
| `connected` | SSE 连接建立，服务端接受请求       | ✅            | 4.1.1      |
| `progress`  | 每个处理步骤开始时                 | ✅            | 4.1.2      |
| `day_ready` | 每一天行程生成完成时               | ✅ **最关键** | 4.1.3      |
| `done`      | 全部完成                           | ✅            | 4.1.4      |
| `error`     | 生成失败（可恢复或不可恢复）       | ✅            | 4.1.5      |
| `warning`   | 非致命警告（如部分景点信息未确认） | ⚠️ 建议处理   | 4.1.6      |

> **重要变更（v1.0.1）**：`accommodation_ready` 事件已合并到 `day_ready` 事件，不再有独立事件。住宿信息作为 `day_ready` 事件中 `day.accommodation` 字段返回（仅每个城市第一天包含此字段）。

#### 4.1.1 `connected` 事件

**触发时机**：SSE 连接建立，服务端接受请求

**数据结构**：

#### 4.1.5 `error` 事件

**触发时机**：生成过程中发生错误（可能是可恢复的，也可能是不可恢复的）

**数据结构**：

```json
event: error
data: {
  "code": 50001,
  "message": "部分景点预约信息查询失败",
  "recoverable": true,
  "partialTripId": "trip_xyz789",
  "failedStep": "booking_info_fetch",
  "suggestion": "可继续查看已生成的行程，或稍后重新生成"
}
```

**字段说明**：

| 字段            | 类型    | 说明                                                                  |
| --------------- | ------- | --------------------------------------------------------------------- |
| `code`          | number  | 错误码（见 API 文档第 8 章）                                          |
| `message`       | string  | 错误描述                                                              |
| `recoverable`   | boolean | 是否可恢复（`true` = 可恢复，继续等待；`false` = 不可恢复，停止生成） |
| `partialTripId` | string  | 若已有部分结果，返回 tripId                                           |
| `failedStep`    | string  | 失败的步骤名称                                                        |
| `suggestion`    | string  | 给用户的建议                                                          |

**前端处理规范**：

| `recoverable` | 前端处理                                                               |
| ------------- | ---------------------------------------------------------------------- |
| `true`        | 显示警告提示（非阻断），继续等待后续事件；提示文案来自 `suggestion`    |
| `false`       | 停止生成，显示错误页面；若有 `partialTripId`，提供"查看已生成部分"按钮 |

#### 4.1.6 `warning` 事件

**触发时机**：生成过程中发生非致命问题（不影响继续使用）

**数据结构**：

```json

  "code": 30001,
  "message": "Day 3 的部分景点营业时间未确认，建议出行前核实",

}
```

**前端处理规范**（建议处理，非强制）：

| 处理方式               | 说明                                           |
| ---------------------- | ---------------------------------------------- |
| **方式一：静默记录**   | 将警告存储到前端状态，用户点击"查看警告"时展示 |
| **方式二：轻提示**     | 在生成中页面底部显示黄色警告条（可关闭）       |
| **方式三：结果页提示** | 在行程结果页顶部显示警告摘要                   |

**推荐方式**：方式二（轻提示）+ 方式三（结果页提示）

**警告提示视觉规范**：


颜色：黄色（`#FAAD14`），位置：生成中页面底部（固定），可手动关闭。

### 4.2 前端状态机（详细版）

```
状态定义：

  idle:
    - 初始状态
    - 进入生成中页面时处于此状态
    - 操作：发起 SSE 连接，转移到 connecting

  connecting:
    - 正在建立 SSE 连接
    - 显示："正在连接..." 动画
    - 超时：5 秒未收到 connected 事件 → 转移到 error
    - 收到 connected 事件 → 转移到 streaming
    - 连接失败 → 转移到 error

  streaming:
    - 生成进行中
    - 可收到以下事件：
      - progress：更新进度条和等待文案（含 subMessage）
      - day_ready：渲染该天卡片（渐进渲染），检查并渲染住宿卡片
      - warning：记录警告，显示轻提示（非阻断），继续等待
      - error：根据 `recoverable` 字段决定是显示警告继续等待，还是转移到 error
      - done：转移到 all_complete
    - 用户主动取消：调用取消接口，转移到 idle

  all_complete:
    - 全部生成完成（收到 done 事件）
    - 操作：显示"生成完成"提示，3 秒后自动跳转（或用户手动跳转）
    - 转移：跳转到行程结果页

  error:
    - 生成失败
    - 显示错误信息和恢复选项
    - 若有 partialTripId，提供"查看已生成部分"按钮
    - 转移：用户选择"重试" → idle；用户选择"查看已生成部分" → 行程结果页

  timeout:
    - SSE 连接超时（120 秒未收到 done 事件）
    - 显示："生成超时，可查看已生成部分"
    - 转移：同 error
```

**状态转换表**：

| 当前状态       | 事件/条件                           | 下一状态                                                      |
| -------------- | ----------------------------------- | ------------------------------------------------------------- |
| `idle`         | 用户进入生成中页面                  | `connecting`                                                  |
| `connecting`   | 收到 `connected`                    | `streaming`                                                   |
| `connecting`   | 连接超时（5s）                      | `error`                                                       |
| `connecting`   | 连接失败（网络错误）                | `error`                                                       |
| `streaming`    | 收到 `progress`                     | `streaming`（更新进度条 + 文案 + subMessage）                 |
| `streaming`    | 收到 `day_ready`                    | `streaming`（渲染天数卡片 + 住宿卡片 + 更新 `completedDays`） |
| `streaming`    | 收到 `warning`                      | `streaming`（记录警告，显示轻提示，继续等待）                 |
| `streaming`    | 收到 `error` 且 `recoverable=true`  | `streaming`（显示警告提示，继续等待）                         |
| `streaming`    | 收到 `error` 且 `recoverable=false` | `error`（停止生成，显示错误页面）                             |
| `streaming`    | 收到 `done`                         | `all_complete`                                                |
| `streaming`    | 用户点击"取消"                      | `idle`（调用取消接口后）                                      |
| `streaming`    | 连接断开（网络问题）                | `streaming`（SSE 自动重连，显示"连接断开，正在重连..."提示）  |
| `streaming`    | 超时（120s 未收到 `done`）          | `timeout`（显示超时提示，提供查看已生成部分选项）             |
| `all_complete` | 用户点击"查看攻略"或 3 秒自动       | 跳转行程结果页                                                |
| `error`        | 用户点击"重试"                      | `idle`                                                        |
| `error`        | 用户点击"查看已生成部分"            | 跳转行程结果页                                                |
| `timeout`      | 用户点击"查看已生成部分"            | 跳转行程结果页                                                |
| `timeout`      | 用户点击"继续生成"                  | `idle`（重新发起 SSE 连接）                                   |

### 4.3 每天到达时的 DOM 更新规则

**核心原则**：收到 `day_ready` 事件后，立即更新 UI，不用等全部完成。

**更新步骤**：

```
1. 收到 day_ready 事件
   data: {
     "dayIndex": 2,
     "day": {
       "dayIndex": 2,
       "date": "2026-07-02",
       "dayType": "city_exploration",
       "cityName": "长沙",
       "title": "Day 2 · 长沙深度游",
       "timeline": [ ... ],
       "accommodation": null,   // 非城市第一天时为 null（仅当 isFirstDayOfCity=true 时有值）
       "isFirstDayOfCity": false,   // ← 新增字段：是否为该城市的第一天
       "tips": [ ... ]
     }
   }

2. 检查 dayIndex 是否已渲染
   - 若已渲染（重试/重连场景），更新该天卡片
   - 若未渲染，创建新卡片（淡入动画）

3. 更新进度区域：
   - 将该天在"天数列表"中标记为"已完成"（✅）
   - 更新进度条百分比（若 progress 事件已返回 percent 字段）

4. 渲染/更新该天卡片：
   - 卡片标题：{dayIndex} · {date} · {title}
   - 时间轴：渲染 day.timeline 数组（含 location、energyLevel、bookingRequired 等字段）
   - 提示：渲染 day.tips 数组
   - 若 day.isFirstDayOfCity 为 true（仅每个城市第一天为 true），渲染住宿推荐卡片
     - 读取 day.accommodation 字段，渲染主选和备选住宿

5. 页面滚动（可选）：
   - 若用户未手动滚动，自动滚动到最新完成的 day 卡片
   - 若用户已手动滚动（查看之前的天），不自动滚动

6. 动画效果：
   - 新完成的 day 卡片：淡入动画（300ms）
   - 进度条：从旧百分比平滑过渡到新百分比（300ms 缓动）
   - 住宿卡片（若有）：从时间轴下方滑出（200ms）
```

**伪代码描述（非实现）**：

```javascript
// 前端收到 SSE 事件后的处理流程（规格描述，非实现代码）
// 依据：API接口设计规格书_v1.0.0.md 第 5 章

function handleSSEEvent(event) {
  switch (event.type) {
    case "connected":
      // 1. 保存 taskId（用于取消生成）
      const taskId = event.data.taskId;
      saveToState("taskId", taskId);

      // 2. 初始化进度条（0%）
      updateProgressBar(0);

      // 3. 显示预计等待时间
      showEstimatedTime(event.data.estimatedTotalSeconds);

      // 4. 显示"正在生成..."文案
      showMessage(event.data.message);

      // 5. 初始化天级卡片占位区（等待 progress/day_ready 事件）
      initDayPlaceholders();
      break;

    case "progress":
      // 1. 更新进度条（event.data.percent）
      updateProgressBar(event.data.percent);

      // 2. 更新当前步骤文案（event.data.message）
      showMessage(event.data.message);

      // 3. 更新辅助文案（event.data.subMessage，若有）
      if (event.data.subMessage) {
        showSubMessage(event.data.subMessage);
      }

      // 4. 更新预计剩余时间（event.data.estimatedRemainingSeconds）
      if (event.data.estimatedRemainingSeconds) {
        showEstimatedRemainingTime(event.data.estimatedRemainingSeconds);
      }
      break;

    case "day_ready":
      // 1. 获取 dayIndex 和 day 数据
      const { dayIndex, day } = event.data;

      // 2. 检查该天卡片是否已存在（处理重试/重连场景）
      let dayCard = findDayCard(dayIndex);

      if (!dayCard) {
        // 创建新卡片（淡入动画）
        dayCard = createDayCard(day);
        dayCard.classList.add("fade-in");
        appendToDayList(dayCard);
      } else {
        // 更新已存在的卡片
        updateDayCard(dayCard, day);
      }

      // 3. 渲染时间轴（含 location、energyLevel、bookingRequired 等字段）
      renderTimeline(dayCard, day.timeline);

      // 4. 若有住宿信息（仅每个城市第一天有），渲染住宿卡片
      if (day.isFirstDayOfCity) {
        renderAccommodationCard(dayCard, day.accommodation);
      }

      // 5. 渲染提示
      renderTips(dayCard, day.tips);

      // 6. 更新进度区域（标记该天为已完成）
      markDayAsCompleted(dayIndex);

      // 7. 自动滚动到最新卡片（若用户未手动滚动）
      autoScrollIfNeeded(dayCard);

      break;

    case "warning":
      // 1. 记录警告到状态
      addWarning(event.data);

      // 2. 显示轻提示（黄色警告条，可关闭）
      showWarningBar(event.data.message);
      break;

    case "error":
      // 1. 判断是否为可恢复错误
      if (event.data.recoverable) {
        // 可恢复：显示警告提示，继续等待
        showRecoverableErrorWarning(event.data);
      } else {
        // 不可恢复：转移到 error 状态
        transitionTo("error", event.data);
      }
      break;

    case "done":
      // 1. 保存 tripId
      saveToState("tripId", event.data.tripId);

      // 2. 显示"生成完成"提示
      showCompletionMessage(event.data.summary);

      // 3. 3 秒后自动跳转（或用户手动跳转）
      scheduleRedirect(`/trip/${event.data.tripId}`, 3000);
      break;
  }
}
```

### 4.4 错误处理与重试策略

#### 4.4.1 错误分类

| 错误类型                   | 来源                                  | 是否可恢复    | 前端处理                 |
| -------------------------- | ------------------------------------- | ------------- | ------------------------ |
| **网络错误**               | SSE 连接断开                          | ✅ 可恢复     | SSE 自动重连（原生支持） |
| **服务端错误（可恢复）**   | SSE `error` 事件，`recoverable=true`  | ✅ 可恢复     | 显示警告，继续等待       |
| **服务端错误（不可恢复）** | SSE `error` 事件，`recoverable=false` | ❌ 不可恢复   | 显示错误，提供"重试"按钮 |
| **超时**                   | 120 秒未收到 `done`                   | ⚠️ 部分可恢复 | 提供"查看已生成部分"按钮 |

#### 4.4.2 重试策略

**SSE 自动重连**：

- SSE 原生支持断线重连（EventSource API）
- 若使用 `@microsoft/fetch-event-source`（支持 POST），需要手动实现重连逻辑
- 重连间隔：指数退避（1s, 2s, 4s, 8s，最大 30s）

**生成失败重试**：

- 用户点击"重试"按钮后，重新发起 SSE 连接
- 若之前已有部分结果（`partialTripId`），后端继续生成未完成的天数

#### 4.4.3 僵死连接检测

**问题**：SSE 连接成功建立，但服务端长时间不返回任何事件（或 `progress` 事件间隔过长），前端无法判断是"正在处理"还是"服务端已僵死"。

**检测规则**：

| 检测项                                       | 超时时间                              | 处理方式                                            |
| -------------------------------------------- | ------------------------------------- | --------------------------------------------------- |
| 连接建立后无 `connected` 事件                | 10 秒                                 | 判定为网络连接异常，进入 SSE 重连逻辑               |
| 收到 `connected` 后无 `progress`             | 30 秒                                 | 显示"生成启动中，请稍候..."提示，不关闭连接         |
| 收到 `progress` 后下一个 `progress` 间隔过长 | 60 秒                                 | 显示"当前步骤处理较慢..."提示                       |
| 收到任意事件后 120 秒内无任何新事件          | 120 秒（对应 `maxProcessingSeconds`） | 判定为僵死连接，前端主动关闭 SSE 连接，提示用户重试 |

**前端处理规范**：

```
onSSEOpen() {
  this.deadConnectionTimer = setTimeout(() => {
    // 120秒内无任何事件，判定为僵死连接
    this.closeSSEConnection()
    this.showError('生成超时，请重试或查看已生成部分')
  }, 120_000)
}

onSSEMessage(event) {
  clearTimeout(this.deadConnectionTimer)   // 收到任意事件，重置定时器
  this.deadConnectionTimer = setTimeout(..., 120_000)

  if (event.event === 'connected') { ... }
  if (event.event === 'progress') { ... }
  if (event.event === 'day_ready') { ... }
  if (event.event === 'done') {
    clearTimeout(this.deadConnectionTimer)  // 完成后清除定时器
  }
}
```

### 4.5 超时与降级策略

#### 4.5.1 超时定义

| 超时类型     | 时长   | 触发条件                                                                              | 处理方式                          |
| ------------ | ------ | ------------------------------------------------------------------------------------- | --------------------------------- |
| **连接超时** | 5 秒   | SSE 连接未在 5 秒内建立                                                               | 显示"连接失败，正在重试..."       |
| **生成超时** | 120 秒 | 发起 SSE 连接后 120 秒未收到 `done` 事件（对应 API 文档 `maxProcessingSeconds` 参数） | 前端主动关闭连接，显示超时提示    |
| **单步超时** | 30 秒  | 某个 `progress` 步骤超过 30 秒未更新                                                  | 显示"当前步骤较慢，请耐心等待..." |

#### 4.5.2 降级方案（轮询 GET /api/trips/:id）

**触发条件**：SSE 连接失败且重连多次后仍失败

**降级流程**：

```
1. SSE 连接失败（网络不支持 SSE / 代理阻断 / ...）
2. 前端检测到 SSE 不可用
3. 前端改用轮询方式：
   - 调用 GET /trips/generate/status/{taskId}
   - 每 3 秒轮询一次
   - 轮询直到 status = 'completed' 或 'failed'
4. 轮询期间，前端显示"正在生成..." + 轮询进度
5. 完成后，跳转到行程结果页
```

**轮询界面设计**：

```
┌──────────────────────────────────────────────────────┐
│  🎯 正在为你生成专属攻略...                           │
│                                                      │
│  ⚠️ 检测到网络连接不稳定，已切换为轮询模式             │
│                                                      │
│  进度：65%   正在生成 Day 4...                        │
│  [=======         ]                                  │
│                                                      │
│            [ 取消生成 ]                               │
└──────────────────────────────────────────────────────┘
```

> **说明**：MVP 阶段 SSE 兼容性较好（现代浏览器均支持），轮询降级方案优先级较低。
