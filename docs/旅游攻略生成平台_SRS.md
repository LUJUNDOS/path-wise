## 1. 项目概述

### 1.1 背景与问题陈述

现有旅游攻略内容存在以下系统性缺陷：

| 问题维度           | 具体表现                               |
| ------------------ | -------------------------------------- |
| **内容同质化**     | 小红书、马蜂窝攻略高度重复，缺乏个性化 |
| **时序割裂**       | 攻略以"城市"为维度，不考虑大交通衔接   |
| **静态化**         | 内容不反映实时状态（闭馆、客满、排队） |
| **人群无差异**     | 带娃家庭与背包客得到相同推荐           |
| **多城市规划混乱** | 跨城市行程的中转日完全空白             |

### 1.2 产品定位

> **一句话定位**：面向所有城市出游旅行者的 **AI 旅行生命周期生成平台**——不是攻略，是一份属于你的完整行程剧本。

### 1.3 核心价值主张

- **旅行生命周期（Trip Lifecycle）**：以"天（Day Timeline）"为最小粒度，不以城市为割裂单位
- **大交通感知**：行程规划天然嵌入出发/抵达/中转逻辑
- **人群动态路由**：根据旅行成员构成（婴儿/老人/情侣/家庭）生成差异化攻略
- **实时数据驱动**：接入交通/住宿/景点实时状态，攻略即时可执行

### 1.4 目标与成功指标

| 目标             | 衡量指标            | 目标值（上线半年） |
| ---------------- | ------------------- | ------------------ |
| 用户完成攻略生成 | 攻略生成完成率      | ≥ 85%              |
| 攻略质量满意度   | 用户评分 ≥ 4 星比例 | ≥ 70%              |
| 多城市支持覆盖   | 支持城市数量        | ≥ 50 座主要城市    |
| 行程执行率       | 用户按攻略出行比例  | ≥ 60%              |

## 2. 用户角色与核心场景

### 2.1 用户角色（User Personas）

#### P1：都市周末游客 · 小李（25岁，单身，上海）

- **特征**：工作日忙碌，只有周末/小长假出行，预算有限，追求效率
- **痛点**：不知道怎么规划，不想花大量时间研究
- **核心需求**：快速生成 2-3 天攻略，性价比优先，交通一键搞定

#### P2：家庭出游者 · 张先生（35岁，已婚，2岁孩子+老人同行）

- **特征**：需要考虑全家人的行动能力，住宿选择复杂
- **痛点**：带老人/孩子的攻略完全不同，排队、距离、房型都是痛
- **核心需求**：自动识别家庭构成，优化行程节奏，智能推荐亲子房型

#### P3：深度游旅行者 · 陈女士（28岁，情侣出行，摄影爱好者）

- **特征**：追求非同质化体验，有一定预算，希望深度了解当地文化
- **痛点**：所有攻略都一样，不想去网红打卡点，想找隐秘好地方
- **核心需求**：多标签偏好过滤，文化/美食/摄影维度深化推荐

#### P4：商旅+旅游结合用户 · 王总（42岁，频繁出差）

- **特征**：出差城市顺带旅游，时间碎片化，预算充裕
- **痛点**：出差之余想游玩但没时间规划，行程高度不确定
- **核心需求**：半天攻略、机场/高铁站附近精华游、快速输出

### 2.2 核心用例（Use Cases）

#### UC-01：基础行程生成

```
前置条件：用户已进入攻略生成页面
主要流程：
  1. 用户输入出发地（城市）
  2. 用户输入目的地城市（支持多个，每个可设定停留天数：1天=途经/打卡，多天=深度游玩）
  3. 用户选择出发日期 + 出发时间段（上午/中午/下午）
  4. 用户设置旅行偏好（预算/节奏/成员构成）
  5. 点击"生成攻略"
  6. 系统调用 AI Agent，查询实时交通数据，生成 Trip Lifecycle
  7. 前端以 Day-by-Day 时间轴展示攻略
  8. 用户可微调单天内容
后置条件：攻略已保存，可分享/导出
```

#### UC-02：多城市中转日处理

```
触发条件：destinations 数组长度 ≥ 2
特殊逻辑：
  1. 系统识别"最后游玩日 + 出发时间"组合
  2. 根据大交通出发时间动态调整当日活动
  3. 计算到达交通枢纽所需时间（地铁/打车两种方案）
  4. 无缝拼接"目的地第一天"的晚间攻略
  5. 中转日提示：行李托管位置、安检时间预留、出发前消费建议
```

#### UC-03：智能住宿匹配

```
触发条件：旅行成员包含老人/婴儿 或 旅行人数 ≥ 3
判定逻辑：
  1. 统计人口结构：成人数、儿童数(年龄)、老人数
  2. 自动推导所需床型（大床/双床/连通房/套房）
  3. 应用噪音区域黑名单（周边有夜市/酒吧街的酒店降权）
  4. 主选房 + Backup Hotel（满房时自动备选）
  5. 输出含价格、地图位置、房型说明的住宿卡片
```

#### UC-04：攻略实时调优

```
触发条件：用户已有攻略，发生异常（景点闭馆/天气变化/交通延误）
执行逻辑：
  1. 系统主动提醒或用户主动触发调优
  2. AI 识别受影响的时间段
  3. 从备选库拉取同等级替代方案
  4. 重新规划受影响的半天行程
  5. 更新攻略，保持整体行程连贯性
```

> **M01 设计决策（2026-06-20）**：
>
> 1. **途经城市统一为「目的地」**：MVP 阶段不区分「目的地」和「途经城市」，统一为「目的地」概念。用户通过设定停留天数自然区分深游（多天）与途经（1 天）。原 M01-F03 已合并至 M01-F02。阶段二可考虑按停留天数自动生成标签（如 1 天显示「途经」）。
> 2. **不做完整对话式 UI**：MVP 阶段不做类 ChatGPT 的对话式 UI，采用「表单 + 轻量智能引导」方案（智能推荐目的地、智能天数建议、生成前确认弹窗）。对话式调优（生成后通过对话微调行程）推迟到阶段二。详见前端交互设计规格书 v1.0.9 第 3.11 节。

#### M07：用户输入冲突检测与提示模块

> **背景**：用户可能输入矛盾的需求组合（如「穷游预算 + 精品酒店」），系统需在生成前检测并提示，避免生成不合理攻略。

| 功能 ID | 功能名称              | 优先级 | 说明                                                                |
| ------- | --------------------- | ------ | ------------------------------------------------------------------- |
| M07-F01 | 预算-住宿冲突检测     | P0     | `budget=economy` + `accommodation=boutique/luxury` → 提示可能超预算 |
| M07-F02 | 预算-餐饮冲突检测     | P1     | `budget=economy` + 高端餐厅偏好 → 提示推荐本地小吃为主              |
| M07-F03 | 人群-节奏冲突检测     | P0     | 有老人/幼儿 + `pace=intensive` → 提示建议调整为适中节奏             |
| M07-F04 | 幼儿-交通冲突检测     | P1     | 有 <3 岁幼儿 + 高铁 > 5h 无航班替代 → 提示是否考虑飞机              |
| M07-F05 | 节奏-人群冲突检测     | P1     | `pace=intensive` + 有老人或幼儿 → 提示强节奏对老幼较辛苦            |
| M07-F06 | 预算-住宿反向冲突检测 | P1     | `budget=luxury` + `accommodation=hostel` → 提示是否确认选择         |
| M07-F07 | 冲突提示交互          | P0     | 提示不阻断生成，用户可坚持原选择，LLM 做折中方案                    |

**冲突提示交互设计**：

```
用户点击「生成攻略」按钮
        ↓
前端调用 POST /api/trips/validate（仅做冲突检测，不触发生成）
        ↓
┌────────────────────────────────────────────┐
│  无冲突                                    │
│  → 正常跳转生成等待页，发起 SSE 生成流程      │
└────────────────────────────────────────────┘
        ↓
┌────────────────────────────────────────────┐
│  有冲突                                    │
│  → 弹出提示弹窗（非阻断，可关闭）           │
│     「检测到以下可能不合理的地方：」             │
│     • 穷游预算下选择精品酒店，可能超预算        │
│       [保持当前选择]  [调整为经济型酒店]       │
│     • 同行有老人，高强度节奏可能较辛苦          │
│       [保持当前选择]  [调整为适中节奏]        │
│                                             │
│     [坚持生成（按比例折中）]  [返回修改]      │
└────────────────────────────────────────────┘
        ↓
用户选择「坚持生成」→ 将冲突标记写入 request_snapshot.conficts
        ↓
LLM 生成时读取 conflicts 字段，做折中方案
        ↓
攻略生成完成后，在首页注明：
  「注意：您的选择存在以下冲突，已为您做折中处理：...」
```

## 4. 非功能性需求

### 4.1 性能需求

| 指标             | 要求                            | 说明                            |
| ---------------- | ------------------------------- | ------------------------------- |
| **攻略生成时延** | P50 ≤ 15s，P99 ≤ 30s            | AI 生成耗时，需流式输出降低感知 |
| **页面首屏加载** | ≤ 2s（4G网络）                  | 前端优化，CDN 加速              |
| **交通数据查询** | ≤ 3s                            | 独立查询接口，与攻略生成并行    |
| **并发用户支持** | 初期 1,000 DAU，峰值 5,000 并发 | 节假日流量峰值 5x 基准          |
| **数据库读取**   | P99 ≤ 100ms                     | 城市知识库、景点库查询          |

### 4.2 可用性与稳定性

| 指标               | 要求                                   |
| ------------------ | -------------------------------------- |
| **系统可用性**     | SLA ≥ 99.5%（年度）                    |
| **攻略生成成功率** | ≥ 99%（AI 故障降级为模板攻略）         |
| **数据备份**       | 用户攻略数据每日全量备份，增量实时备份 |
| **故障恢复**       | RTO ≤ 30min，RPO ≤ 1h                  |

### 4.3 安全性需求

| 维度                | 要求                                                 |
| ------------------- | ---------------------------------------------------- |
| **接口安全**        | 所有 API 使用 HTTPS + JWT 鉴权                       |
| **数据隐私**        | 用户位置、行程数据不对外共享，符合《个人信息保护法》 |
| **防刷保护**        | 攻略生成接口限流（单用户每分钟 ≤ 3 次）              |
| **第三方 API 安全** | 交通/酒店 API Key 服务端存储，不暴露前端             |
| **输入校验**        | 所有用户输入服务端二次校验，防 Prompt Injection      |

### 4.4 可扩展性需求

| 维度               | 要求                                             |
| ------------------ | ------------------------------------------------ |
| **城市知识库扩展** | 新增城市无需修改核心代码，注册即生效（工厂模式） |
| **AI 模型可替换**  | LLM 调用层抽象，支持切换 GPT-4/Claude/国内大模型 |
| **多端适配**       | Web 端优先，预留移动端 API 接口                  |

### 6.1 旅行生命周期生成流程（主流程）

```

  ▼

         ▼

  ├── 出发段：departure → destinations[0]
  └── 各中转段：destinations[n] → destinations[n+1]
  │
  ▼
[Trip Lifecycle 时间轴构建]
  │
  ├── Day 1：始发移动日

  │   ├── 顺路串联算法（地理位置优化）
  │   └── 人群标签过滤（老人/孩子/偏好）
  │
  ├── Day N（如有下一城市）：中转衔接日
  │   ├── 查询离港大交通时间
  │   ├── 计算安全到达交通枢纽的最晚出发时间
  │   ├── 推断可用活动时间窗口
  │   ├── 强约束：过滤高体力/远距离项目
  │   ├── 推荐枢纽周边伴手礼/轻活动

      ├── 判断用户是否勾选「预定返程」
      ├── 若勾选：查询末站城市→出发地返程大交通（同出发日缓冲时间逻辑）
```

### 6.2 中转日算法伪代码逻辑

```
function buildTransitDay(currentCity, nextCity, userProfile):

  // 1. 获取出发大交通信息
  transport = getTransport(currentCity, nextCity)
  departureTime = transport.departureTime  // e.g. 16:00

  // 2. 计算最晚离开市区时间
  transferType = transport.type  // "high_speed_rail" | "normal_train" | "flight" | "bus"
  bufferTime = (transferType == "flight") ? 120 : (transferType == "bus") ? 60 : 90  // 分钟
  // high_speed_rail / normal_train: 90min
  // bus: 60min (提前1小时)
  stationTransitTime = getTransitToStation(currentCity, transport.station)  // 地铁/打车时间
  latestDepartureFromCity = departureTime - bufferTime - stationTransitTime
  // e.g. 16:00 - 90min - 45min = 13:45 最晚从市区出发

  // 3. 可用活动时间窗口
  activityWindow = {
    start: "morning_start",    // e.g. 09:00
    end: latestDepartureFromCity  // e.g. 13:45
  }

  // 4. 约束条件
  constraints = [
    "distance_from_station <= 5km",  // 活动地点不能离站太远
    "physical_intensity <= LOW",     // 禁止高体力活动
    "duration_per_activity <= 2h",   // 单个活动不超过2小时
    "include_luggage_storage_tip"    // 必须包含行李寄存提示
  ]

  // 5. 到达目的地后衔接
  arrivalTime = transport.arrivalTime  // e.g. 19:00
  eveningPlan = getCityRules(nextCity).getEveningPlan(arrivalTime)

  return buildDayPlan(activityWindow, constraints, eveningPlan)
```

### 6.3 房型智能判定规则

```
人口结构输入：
  adults: 2, children: [{age: 2}], elders: 1

判定逻辑：
  总入住人数 = 2（成人）+ 1（2岁）+ 1（老人）= 4人（实际2大1老，婴儿计0.5）

  规则树：
  ├── 总人数 ≥ 4 且含老人 → 优先考虑套房或两房一厅
  ├── 含婴儿 → 需要婴儿床（crib），过滤无婴儿床房型
  ├── 含老人 → 优先低楼层、无台阶、靠近电梯房间
  ├── 噪音判定 → 老人存在 → 应用噪音黑名单
  └── 最终推荐：
        主选：豪华家庭套房（2大床/1大床+沙发床，含婴儿床）
        Backup：两间相邻标准大床房（如套房满房时）
```

## 7. 未来扩展预留

### 7.1 自驾游独立模块（Reserved）

**决策**：MVP 及阶段二均不支持自驾游场景，自驾游作为独立模块在未来阶段实现。

**预留设计**：

- `DestinationConfig.transportTo` 枚举预留 `"driving"` 值，MVP 阶段前端隐藏该选项
- 自驾游场景与现有 Trip Lifecycle 模型差异较大，建议实现为独立生成器模块，不与高铁/飞机逻辑混用

**自驾游独立模块设计要点（未来实现时参考）**：

```
自驾游生成器（独立模块）：
  输入：出发地 + 终点城市（或途经城市列表）+ 总天数
  核心逻辑差异：
    - 不以「城市」为单位安排住宿，而以「当日驾驶终点」为单位
    - 途经点可能是小镇/服务区/景点，而非城市
    - 每日驾驶距离约束（建议 ≤ 500km/天）
    - 开车本身消耗体力，到达后不宜安排高强度活动
    - 住宿安排在每日驾驶结束后的最近城市
  输出：公路旅行行程（与标准 Trip Lifecycle 输出格式对齐）
```

**触发条件**：阶段三（商业化后）根据用户需求评估是否实现。
