## 1. 设计原则与数据库选型

### 1.1 设计原则

| 原则               | 说明                                                                         |
| ------------------ | ---------------------------------------------------------------------------- |
| **最小化冗余**     | 核心业务数据以范式化方式存储，避免更新异常                                   |
| **JSONB 灵活扩展** | 攻略时间轴、用户偏好等半结构化数据用 JSONB 存储，不强制表结构                |
| **缓存前置**       | 高频读取数据（POI、路线、天气）优先从 Redis 读取，不穿透到 PostgreSQL        |
| **软删除**         | 用户数据一律软删除（`deleted_at` 字段），满足《个人信息保护法》30 天删除要求 |
| **审计追踪**       | 所有核心表包含 `created_at`、`updated_at`，攻略修改记录完整历史              |
| **多租户预留**     | 表结构预留 `tenant_id` 字段，支持未来 SaaS 多租户扩展                        |

### 1.2 数据库选型矩阵

| 数据库                          | 用途               | 阶段   | 说明                                       |
| ------------------------------- | ------------------ | ------ | ------------------------------------------ |
| **PostgreSQL 16**               | 主业务数据         | MVP +  | 用户、攻略、日计划、住宿推荐               |
| **Redis 7**                     | 缓存 + 队列 + 限流 | MVP +  | 高德缓存、LLM 结果缓存、BullMQ、Rate Limit |
| **Elasticsearch 8**             | 全文搜索           | 阶段二 | 城市/景点/餐厅模糊搜索                     |
| **pgvector（PostgreSQL 插件）** | 语义向量检索       | 阶段三 | 城市知识库 Embedding 检索                  |
| **本地 JSON 文件**              | 城市知识库         | MVP    | 静态 POI 数据，零成本，启动快              |

> **MVP 阶段数据库范围**：PostgreSQL + Redis + 本地 JSON。Elasticsearch 和 pgvector 在阶段二/三引入。

## 3. PostgreSQL — 主库表结构

### 3.1 用户表 `users`

```sql
CREATE TABLE users (
  id            UUID          PRIMARY KEY DEFAULT gen_random_uuid(),
  phone         VARCHAR(20)   UNIQUE,                    -- 手机号（主要登录方式）
  email         VARCHAR(255)  UNIQUE,                    -- 邮箱（备用登录）
  nickname      VARCHAR(64)   NOT NULL DEFAULT '旅行者',
  avatar_url    TEXT,
  -- 默认偏好（可被单次攻略覆盖）
  default_budget_level   VARCHAR(20) DEFAULT 'comfort',  -- economy / comfort / luxury
  default_pace_level     VARCHAR(20) DEFAULT 'moderate', -- intensive / moderate / relaxed
  default_accommodation  VARCHAR(30) DEFAULT 'any',      -- hostel / chain_hotel / boutique / any
  -- 账户状态
  status        VARCHAR(20)   NOT NULL DEFAULT 'active', -- active / banned / unverified
  -- 审计字段
  last_login_at TIMESTAMPTZ,
  created_at    TIMESTAMPTZ   NOT NULL DEFAULT NOW(),
  updated_at    TIMESTAMPTZ   NOT NULL DEFAULT NOW(),
  deleted_at    TIMESTAMPTZ                              -- 软删除，30 天后物理删除
);

COMMENT ON TABLE  users                 IS '用户账户表';
COMMENT ON COLUMN users.default_budget_level IS 'economy=经济 comfort=舒适 luxury=豪华';
```

**字段约束说明**

| 字段                      | 约束                   | 说明                                  |
| ------------------------- | ---------------------- | ------------------------------------- |
| `id`                      | UUID PK                | 避免自增 ID 泄露用户量                |
| `phone` / `email`         | UNIQUE，至少有一个非空 | 应用层校验                            |
| `deleted_at`              | 软删除                 | 置为非空后，30 天定时任务执行物理删除 |
| `default_budget_level` 等 | 有默认值               | 用户未填写偏好时使用系统默认          |

> **平台 LLM API Key 管理说明**（对应问题 4 修正）：
>
> - LLM API Key 由**平台统一提供**，不存储在 `users` 表中
> - MVP 阶段：通过**服务端环境变量**配置（`DEEPSEEK_API_KEY`、`GLM_API_KEY` 等）
> - 阶段二：可新建 `platform_config` 表支持后台管理（见本节末尾）
> - 调用优先级：先读环境变量，若需多租户隔离再读 `platform_config`

### 3.2 攻略主表 `trips`

```sql
CREATE TABLE trips (
  id              UUID          PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id         UUID          NOT NULL REFERENCES users(id),
  -- 基础信息
  title           VARCHAR(200)  NOT NULL,                -- 系统生成标题，如"长沙3天2夜深度游"
  status          VARCHAR(30)   NOT NULL DEFAULT 'generating',
                                                         -- generating / completed / partial / failed
  -- 请求参数快照（完整保留，用于重新生成）
  -- ⚠️ 注意：
  --    1. 偏好数据（预算档位、节奏、住宿偏好等）不单独建 trip_preferences 表，
  --       一律嵌入此 JSONB 字段存储，符合 §1.1「半结构化数据用 JSONB」的设计原则。
  --    2. 返程参数（needsReturnTransport / returnTransportPref）也嵌入此字段，
  --       不单独建表。
  request_snapshot JSONB        NOT NULL,
  -- 行程基本信息
  departure_city  VARCHAR(50)   NOT NULL,
  total_days      SMALLINT      NOT NULL CHECK (total_days > 0 AND total_days <= 30),
  start_date      DATE          NOT NULL,
  end_date        DATE          NOT NULL,
  -- 预算汇总
  total_estimated_cost_cny DECIMAL(10, 2),
  budget_breakdown JSONB,                               -- { transport: 1200, hotel: 2400, dining: 800, ... }
  -- 生成元信息
  llm_provider    VARCHAR(30),                           -- 实际使用的 LLM 提供商
  generation_time_ms INTEGER,                            -- 生成耗时（毫秒）
  incomplete_reason TEXT,                                -- 若 status=partial，记录原因
  -- 可见性
  is_public       BOOLEAN       NOT NULL DEFAULT FALSE,  -- 是否公开（分享）
  view_count      INTEGER       NOT NULL DEFAULT 0,
  -- 审计
  created_at      TIMESTAMPTZ   NOT NULL DEFAULT NOW(),
  updated_at      TIMESTAMPTZ   NOT NULL DEFAULT NOW(),
  deleted_at      TIMESTAMPTZ
);

CREATE INDEX idx_trips_user_id          ON trips(user_id) WHERE deleted_at IS NULL;
CREATE INDEX idx_trips_status           ON trips(status) WHERE deleted_at IS NULL;
CREATE INDEX idx_trips_start_date       ON trips(start_date);

COMMENT ON TABLE  trips                  IS '攻略主表，每条记录对应一次完整的行程攻略';
COMMENT ON COLUMN trips.request_snapshot IS '完整请求参数快照，用于攻略重新生成和展示参数摘要';
COMMENT ON COLUMN trips.status           IS 'generating=生成中 completed=完成 partial=部分完成 failed=失败';
```

> **字段更新权限说明**（对应用户反馈「不实施更新的字段不止这几个」）：
> `trips` 表中以下字段由**系统自动维护**，不开放 API 单独更新：
>
> | 自动管理字段                                                | 维护来源                                   |
> | ----------------------------------------------------------- | ------------------------------------------ |
> | `id`                                                        | 系统生成（UUID）                           |
> | `user_id`                                                   | 创建时绑定，不可变更                       |
> | `status`                                                    | 生成引擎自动更新                           |
> | `request_snapshot`                                          | 创建时写入，重新生成时整体替换             |
> | `total_estimated_cost_cny` / `budget_breakdown`             | 生成引擎计算写入                           |
> | `llm_provider` / `generation_time_ms`                       | 生成引擎写入                               |
> | `incomplete_reason`                                         | 生成引擎在 partial/failed 时写入           |
> | `created_at` / `updated_at` / `deleted_at`                  | 审计触发器自动维护                         |
> | `departure_city` / `total_days` / `start_date` / `end_date` | 由 `request_snapshot` 冗余，随快照同步更新 |
>
> **用户可通过 API 修改的字段仅限**：`title`（重命名攻略）、`is_public`（可见性切换）。

**status 状态流转**

```
generating
    │
    ├──── 成功 ──────▶ completed
    │
    ├──── 部分成功 ──▶ partial（已生成部分天数，用户可继续生成）
    │
    └──── 失败 ──────▶ failed（全部失败）
```

### 3.3 日计划表 `day_plans`

```sql
CREATE TABLE day_plans (
  id              UUID          PRIMARY KEY DEFAULT gen_random_uuid(),
  trip_id         UUID          NOT NULL REFERENCES trips(id) ON DELETE CASCADE,
  -- 位置
  day_index       SMALLINT      NOT NULL CHECK (day_index >= 1),  -- 第几天（从 1 开始）
  date            DATE          NOT NULL,
  -- 类型
  day_type        VARCHAR(30)   NOT NULL,
                                -- transit_departure / city_exploration / transit_transfer / transit_return
  city_name       VARCHAR(50)   NOT NULL,
  title           VARCHAR(200)  NOT NULL,                         -- 如"Day 2 · 长沙探城"
  -- 交通信息（非空时代表含大交通）
  transport_info  JSONB,
  -- 体力分析
  energy_summary  JSONB,                                          -- { totalCost: 350, avgLevel: "MEDIUM" }
  -- 提示与说明
  tips            TEXT[],                                         -- 当天注意事项
  -- 审计
  created_at      TIMESTAMPTZ   NOT NULL DEFAULT NOW(),
  updated_at      TIMESTAMPTZ   NOT NULL DEFAULT NOW(),
  -- 唯一约束：同一攻略同一天只有一条记录
  UNIQUE (trip_id, day_index)
);

CREATE INDEX idx_day_plans_trip_id ON day_plans(trip_id);

COMMENT ON TABLE  day_plans             IS '日计划表，每行对应攻略中的一天';
COMMENT ON COLUMN day_plans.day_type    IS 'transit_departure=始发移动日 city_exploration=深度游玩日 transit_transfer=中转日 transit_return=返程日（仅 needsReturnTransport=true 时使用）';
```

### 3.5 住宿推荐表 `hotel_recommendations`

> 设计决策：住宿推荐按**城市+入住日期**存储，每个城市一条记录（含 primary 和 backup）。  
> 对应 API 规格书第 5 章：住宿数据随城市第一天 `day_ready` 事件推送。

```sql
CREATE TABLE hotel_recommendations (
  id              UUID          PRIMARY KEY DEFAULT gen_random_uuid(),
  trip_id         UUID          NOT NULL REFERENCES trips(id) ON DELETE CASCADE,
  day_plan_id     UUID          NOT NULL REFERENCES day_plans(id),  -- 对应城市第一天
  -- 城市信息
  city_name       VARCHAR(50)   NOT NULL,
  check_in_date   DATE          NOT NULL,
  check_out_date  DATE          NOT NULL,
  nights          SMALLINT      NOT NULL CHECK (nights > 0),
  -- 推荐方案
  primary_option  JSONB         NOT NULL,                           -- 主推住宿，格式见 HotelOption
  backup_option   JSONB         NOT NULL,                           -- 备选住宿
  -- 推荐理由
  recommendation_reason TEXT,
  -- 用户选择（用户可从 primary/backup 中选择，或自定义）
  user_choice     VARCHAR(20)   DEFAULT 'primary',                  -- primary / backup / custom
  custom_hotel    JSONB,                                            -- 用户自定义住宿信息
  -- 估算总价
  estimated_total_cny DECIMAL(10, 2),
  -- 审计
  created_at      TIMESTAMPTZ   NOT NULL DEFAULT NOW(),
  updated_at      TIMESTAMPTZ   NOT NULL DEFAULT NOW(),
  -- 唯一约束：同一攻略同一城市只有一条住宿推荐
  UNIQUE (trip_id, city_name)
);
```

## 4. Redis — 缓存层设计

### 4.1 命名规范

```
格式：{domain}:{entity}:{identifier}[:{subkey}]

示例：
  amap:poi:city:长沙:岳麓山          ← 高德 POI 搜索缓存
  llm:health:deepseek                ← LLM 提供商健康状态

  rate:user:{userId}:trip_generate   ← 用户限流计数器

```

### 4.2 缓存 Key 完整清单

| Key 模式                                    | 数据类型      | TTL              | 说明                            |
| ------------------------------------------- | ------------- | ---------------- | ------------------------------- |
| `amap:poi:search:{city}:{keywords}`         | String (JSON) | 86400s（24h）    | 高德 POI 搜索结果               |
| `amap:poi:detail:{poiId}`                   | String (JSON) | 604800s（7天）   | 高德 POI 详情                   |
| `amap:route:{originHash}:{destHash}:{mode}` | String (JSON) | 3600s（1h）      | 路线规划结果                    |
| `amap:geocode:{addressHash}`                | String (JSON) | 2592000s（30天） | 地址编码结果                    |
| `amap:weather:{cityCode}:{date}`            | String (JSON) | 7200s（2h）      | 城市天气                        |
| `amap:quota:{apiName}:{date}`               | String (Int)  | 到次日 0 点      | 高德 API 配额余量               |
| `llm:health:{provider}`                     | String (JSON) | 86400s（24h）    | LLM 提供商健康状态              |
| `llm:quota:{provider}:{date}`               | String (Int)  | 到次日 0 点      | LLM 调用配额余量                |

| `rate:user:{userId}:trip_generate`          | String (Int)  | 60s              | 攻略生成限流（1分钟内≤3次）     |
| `rate:user:{userId}:api_global`             | String (Int)  | 60s              | 全局 API 限流                   |
| `session:{token}`                           | String (JSON) | 604800s（7天）   | 用户会话（JWT 黑名单/快速验证） |

### 4.3 核心缓存详细设计

#### LLM 提供商健康状态

```
Key:   llm:health:{provider}
       provider ∈ { deepseek, glm, milm, kimi }

Value (JSON)：
{
  "status": "healthy",             // healthy / degraded / unhealthy
  "consecutiveFailures": 0,
  "consecutiveSuccesses": 12,
  "lastFailureAt": null,
  "lastSuccessAt": "2026-06-19T01:00:00+08:00",
  "retryAfter": null,              // degraded 时：恢复时间戳
  "dailyRequests": 156,
  "dailyFailures": 2,
  "updatedAt": "2026-06-19T01:00:00+08:00"
}

TTL: 86400s（每日凌晨 2:00 由定时任务主动重置）
```

#### SSE 任务状态

```
Key:   trip:sse:task:{taskId}
Type:  Hash

Fields:
  status          generating / completed / failed
  progress        0~100（百分比）
  step            当前步骤描述，如 "正在生成第 2 天行程..."
  completedDays   已完成天数
  totalDays       总天数
  tripId          生成完成后写入
  partialTripId   部分完成时写入
  errorMessage    失败时写入

TTL: 7200s（2小时，足够 SSE 连接恢复）
```

### 4.4 BullMQ 队列设计

> 攻略生成是耗时任务，通过 BullMQ（Redis 队列）异步处理，避免 HTTP 超时。

```
队列名：trip-generation

Job 数据结构：
{
  "taskId": "gen_abc123",
  "tripId": "uuid",
  "userId": "uuid",
  "request": { /* TripRequest */ },
  "priority": 1,          // 1=普通, 2=VIP（预留）
  "retries": 0,
  "maxRetries": 2
}

队列配置：
  concurrency: 5           // 同时处理 5 个生成任务
  defaultJobOptions:
    removeOnComplete: true  // 完成后删除 job（日志已写入 PostgreSQL）
    removeOnFail: false     // 失败保留（供排障）

    backoff: { type: 'exponential', delay: 2000 }
```

## 8. 分区与分表策略

### 8.1 `llm_call_logs` 按月分区

```sql
-- 主表（已在第 3.9 节定义）
-- 每月初自动创建下月分区（由定时任务执行）

-- 分区命名规范：llm_call_logs_YYYY_MM
-- 归档策略：超过 12 个月的分区，导出到对象存储（阿里云 OSS）后 detach

-- 自动创建分区的存储过程（定时任务每月 1 日执行）
CREATE OR REPLACE FUNCTION create_next_month_partition()
RETURNS VOID AS $$
DECLARE
  next_month_start DATE := DATE_TRUNC('month', NOW()) + INTERVAL '1 month';
  next_month_end   DATE := next_month_start + INTERVAL '1 month';
  partition_name   TEXT := 'llm_call_logs_' || TO_CHAR(next_month_start, 'YYYY_MM');
BEGIN
  EXECUTE FORMAT(
    'CREATE TABLE IF NOT EXISTS %I PARTITION OF llm_call_logs FOR VALUES FROM (%L) TO (%L)',
    partition_name, next_month_start, next_month_end
  );
END;
$$ LANGUAGE plpgsql;
```

## 9. 数据备份与恢复策略

对应 SRS v1.1.0 第 4.2 节可用性需求（RPO ≤ 1h，RTO ≤ 30min）。

### 9.1 备份计划

| 类型                 | 频率       | 保留时间 | 存储         | 说明                              |
| -------------------- | ---------- | -------- | ------------ | --------------------------------- |
| **全量备份**         | 每日 02:00 | 30 天    | 阿里云 OSS   | `pg_dump` + 压缩，加密存储        |
| **WAL 归档（增量）** | 实时       | 7 天     | 阿里云 OSS   | 基于 `archive_command`，支持 PITR |
| **Redis 快照**       | 每小时 RDB | 24 小时  | 本地 + OSS   | 缓存数据重建成本低，快照即可      |
| **城市知识库 JSON**  | 每次更新时 | 永久     | Git 版本控制 | 知识库文件纳入 Git 管理           |

### 9.2 恢复步骤（RPO ≤ 1h 场景）

```
场景：数据库实例故障，需要恢复

步骤：
  1. 从 OSS 拉取最近一次全量备份 (~5min)
  2. 恢复全量备份到新实例 (~10min)
  3. 应用 WAL 归档，恢复到故障前最近时间点 (~10min)
  4. 更新 DNS/负载均衡指向新实例 (~5min)
  合计：~30min (RTO 达标)

数据丢失窗口：WAL 实时归档，最大丢失 < 5s (RPO 达标)
```

## 10. 数据迁移规范

### 10.1 迁移原则

| 原则               | 说明                                                    |
| ------------------ | ------------------------------------------------------- |
| **向前兼容**       | 新增字段必须有默认值或允许 NULL                         |
| **禁止破坏性迁移** | 删除列/改变列类型需分两步：先废弃（应用层忽略），再删除 |
| **迁移回滚**       | 每次迁移必须有对应的 `down` 脚本                        |
| **小批次执行**     | 大表的 `UPDATE` 操作分批执行（每批 1000 行），避免锁表  |

### 10.2 Prisma 迁移文件命名规范

```
migrations/
 20260619000001_create_users_table/
    migration.sql
 20260619000002_create_trips_table/
    migration.sql

命名规则：
  {timestamp}_{action}_{target}
  action: create / add / modify / drop / rename
  target: {table}_{column_or_index}
```
