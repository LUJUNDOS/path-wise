# 旅游攻略生成平台 · 技术决策备忘录

> **文档版本**：v1.1.0
> **日期**：2026-06-18
> **主题**：API 数据源选型与 MVP 阶段数据获取策略

---

## 一、背景

SRS 文档完成后，对飞猪/携程等 OTA 平台 API 的准入条件和收费模式进行了调研，结论直接影响 MVP 阶段的数据层架构设计。本文档记录调研结论和技术决策。

---

## 二、主流 OTA 平台 API 准入条件

### 2.1 飞猪开放平台（open.alitrip.com）

| 维度           | 结论                                                   |
| -------------- | ------------------------------------------------------ |
| **准入资质**   | 仅支持企业营业执照认证，**不接受个人开发者**           |
| **收费模式**   | 技术服务费（按 API 调用量阶梯计费，需签约后获取报价）  |
| **保证金**     | 酒店/机票类目需缴纳保证金，金额未公开（预估 ¥10,000+） |
| **对接周期**   | 需商务洽谈，周期长（1-3 个月）                         |
| **个人可行性** | ❌ 不可行                                              |

> 备注：飞猪另设有「飞猪 AI 开放平台（FlyAI）」，同样要求企业资质。

### 2.2 携程开放平台（developer.ctrip.com）

| 维度           | 结论                                                   |
| -------------- | ------------------------------------------------------ |
| **准入资质**   | 面向企业合作伙伴，个人开发者无法直接获取生产级 API Key |
| **收费模式**   | 按效果（订单佣金）+ 调用量双重计费，需承诺最低交易流水 |
| **个人可行性** | ❌ 不可行（需通过第三方服务商间接接入，成本高）        |

### 2.3 高德地图开放平台（lbs.amap.com）✅

| 维度           | 结论                                              |
| -------------- | ------------------------------------------------- |
| **准入资质**   | ✅ 支持个人开发者认证                             |
| **免费额度**   | Web 服务 API：**5,000 次/日**；JS API：无限制     |
| **覆盖能力**   | POI 搜索、路线规划、地理编码/逆地理编码、距离计算 |
| **个人可行性** | ✅ **非常友好，MVP 核心数据来源**                 |

## 四、分阶段演进路线

### 阶段一：MVP（0-3 个月）· 0 成本验证

**目标**：验证 Trip Lifecycle 核心价值，不触碰 OTA 官方 API

| 模块              | 数据来源                       | 成本 |
| ----------------- | ------------------------------ | ---- |
| 景点 POI          | 高德地图 API（免费额度）       | ¥0   |
| 路线规划/距离计算 | 高德地图 API（免费额度）       | ¥0   |
| 高铁查询          | 12306 非官方接口（开源）       | ¥0   |
| 酒店展示          | 高德 POI 搜索，跳转购买        | ¥0   |
| 天气              | 和风天气 API（免费层）         | ¥0   |
| 城市知识库        | 手工维护 JSON，存入 PostgreSQL | ¥0   |

**风险**：12306 非官方接口稳定性无保障，高德免费额度在用户量增长后可能不足。

### 阶段二：功能完善（3-6 个月）· 引入付费聚合 API

**触发条件**：日活 > 100，或高德免费额度频繁触顶

| 模块     | 数据来源                                            | 预估成本    |
| -------- | --------------------------------------------------- | ----------- |
| 高铁查询 | 聚合数据平台（极速数据/万维易源）                   | ¥100-300/月 |
| 机票查询 | 携程 Affiliate API（如有公司主体）或 Skyscanner API | 按调用量    |
| 高德超量 | 购买高德商用授权                                    | ¥5,000/年起 |
| 酒店详情 | 接 Booking.com API（申请个人开发者）                | 免费层优先  |

### 阶段三：商业化（12 个月+）· 官方 API 接入

**触发条件**：有稳定收入，注册公司主体

| 动作          | 说明                             |
| ------------- | -------------------------------- |
| 注册公司主体  | 以企业身份入驻飞猪/携程开放平台  |
| 接入官方 API  | 高铁/机票/酒店全链路预订闭环     |
| 申请 OTA 佣金 | 通过平台分佣变现（核心商业模式） |
| 高德商用授权  | 购买企业版，去除额度限制         |

## 五、关键技术决策记录

### 决策 1：MVP 阶段不接入任何 OTA 官方 API

**理由**：

- 个人开发者无准入资格，强行对接存在合规风险
- 商务对接周期长，与 MVP 快速验证目标冲突
- 预订闭环可在阶段三再补齐，MVP 先做「攻略生成 + 跳转购买」

**风险**：无法做实时房态/余票查询，攻略中的预订链接可能是过期的。缓解方案：在攻略中标注「价格/房态以实际为准」。

### 决策 2：高德地图 API 作为核心 POI/路线数据源

**理由**：

- 个人认证免费额度充足（5,000 次/日 = 支撑 ~500 个攻略生成/天）
- 国内 POI 覆盖最完整，景点/餐厅数据质量高
- 路线规划 API 直接支持「多途经点优化」，天然契合顺路串联算法

**监控指标**：每日 API 调用量，提前 7 天预警额度耗尽。

### 决策 3：12306 非官方接口仅用于 MVP

**理由**：

- 无官方开放 API，非官方接口是唯一可行方案
- 稳定性风险高，需在阶段二切换为付费聚合平台
- 在代码中预留 Adapter 接口，切换数据源时无需修改业务逻辑

### 决策 4：机票/高铁/酒店/餐厅 MVP 阶段只做「展示 + 跳转」，不接入实时预订 API

**结论**：

- 攻略中展示酒店名称、位置、类型标签，**不展示实时价格**（标注「价格以实际为准」）
- 攻略中展示高铁车次号、出发/到达时间，跳转 12306 购票页
- 餐厅/景点只展示名称、位置、营业时间，不展示实时排队状态
- 所有预订类操作均通过 **Deep Link 跳转** 到 OTA 平台完成，不在本平台闭环

**不需要的 API**：
| 场景 | 是否需要官方 API | 替代方案 |
|------|----------------|---------|
| 酒店展示 | ❌ 不需要 | 高德 POI 搜索 |
| 酒店实时价格 | ❌ 不需要 | 不展示具体数字，写「以实际为准」 |
| 高铁班次查询 | ✅ 需要（车次/时间） | 12306 非官方接口 |
| 高铁实时票价 | ❌ 不需要 | 跳转 12306 查看 |
| 机票班次 | ⚠️ MVP 跳过 | 阶段二再接入 |
| 餐厅推荐 | ❌ 不需要 | 高德 POI + 城市知识库 |

**跳转链接格式**：

```
携程酒店：https://hotels.ctrip.com/hotel/{cityCode}.html
飞猪酒店：https://travel.fliggy.com/hotel/search-{cityName}
12306 购票：https://www.12306.cn/index/
```

**商业化路径**：
即使用户通过跳转完成购买，也可申请 OTA Affiliate 计划拿佣金（携程/飞猪均支持个人申请），无需接入官方 API。
