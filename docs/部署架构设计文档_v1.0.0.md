## 1. 概述

### 1.1 文档目的

本文档定义 PATH-WISE 旅游攻略生成平台的部署架构，包括：

- 基础设施选型
- 容器化方案
- CI/CD 流水线
- 监控与日志策略
- 灾难恢复方案

### 1.2 部署原则

| 原则                     | 说明                                 |
| ------------------------ | ------------------------------------ |
| **基础设施即代码 (IaC)** | 所有环境配置用代码管理（Terraform）  |
| **不可变基础设施**       | 部署时替换整个容器，而非更新现有容器 |
| **蓝绿部署**             | 新版本部署到绿色环境，验证后切换流量 |
| **自动化一切**           | 构建、测试、部署全程自动化           |
| **环境一致性**           | 开发、测试、生产环境尽量一致         |

## 3. 基础设施选型

### 3.1 推荐方案：国内云服务商

**推荐理由**：

- ✅ 合规（ICP 备案、数据安全法）
- ✅ 低延迟（国内用户访问速度快）
- ✅ 成本低（相比 AWS/GCP）
- ✅ 中文支持（技术支持沟通方便）

### 3.2 云服务商对比

| 服务商       | 优势                   | 劣势                   | 推荐场景              |
| ------------ | ---------------------- | ---------------------- | --------------------- |
| **阿里云**   | 生态完整、文档丰富     | 价格中等               | ✅ 推荐（综合最优）   |
| **腾讯云**   | 价格便宜、微信生态集成 | 生态不如阿里云         | ✅ 推荐（成本敏感）   |
| **华为云**   | 国企背景、安全可靠     | 生态较弱               | ⚠️ 备选               |
| **AWS 中国** | 国际品牌、功能强大     | 价格高、需要 ICCP 资质 | ❌ 不推荐（MVP 阶段） |

### 3.3 推荐配置（MVP 阶段）

#### 3.3.1 生产环境最小配置

```yaml
# 前端（静态资源）
部署平台: 阿里云 OSS + CDN
  - 存储空间: 50 GB
  - CDN 流量: 500 GB/月
  - 成本: ~¥60/月

# 后端 API
部署平台: 阿里云 ECS（弹性计算服务）
  - 实例规格: ecs.g7.large（2 vCPU, 8 GB 内存）
  - 实例数量: 2（主备）
  - 操作系统: Ubuntu 22.04 LTS
  - 成本: ~¥300/月

# 数据库
部署平台: 阿里云 RDS for PostgreSQL
  - 实例规格: rds.pg.g2.large（2 vCPU, 8 GB 内存）
  - 存储空间: 100 GB SSD
  - 备份: 自动每日备份，保留 7 天
  - 成本: ~¥400/月

# 缓存
部署平台: 阿里云 Redis
  - 实例规格: redis.r2.large（1 vCPU, 2 GB 内存）
  - 成本: ~¥100/月

# 域名 & SSL
域名: path-wise.com（举例）
SSL 证书: 阿里云免费证书（或 Let's Encrypt）
成本: ~¥55/年（域名） + ¥0（SSL）

# 总成本估算（MVP 阶段）
月度成本: ~¥860/月（¥10,320/年）
```

#### 3.3.2 成本优化建议

| 策略               | 说明                           | 节省成本           |
| ------------------ | ------------------------------ | ------------------ |
| **预留实例**       | 包年包月（相比按量付费省 30%） | ~¥3,000/年         |
| **Spot 实例**      | 测试环境用 Spot 实例           | ~¥1,000/年         |
| **CDN 缓存优化**   | 减少回源流量                   | ~¥500/年           |
| **数据库只读实例** | 读写分离（读多写少场景）       | 性能提升，成本不变 |

### 4.2 Dockerfile 设计

#### 4.2.1 前端 Dockerfile（多阶段构建）

```dockerfile
# ========== 阶段 1：构建 ==========
FROM node:22-alpine AS builder

# 设置工作目录
WORKDIR /app

# 复制依赖声明文件
COPY package*.json pnpm-lock.yaml ./

# 安装依赖（使用 pnpm）
RUN npm install -g pnpm && pnpm install --frozen-lockfile

# 复制源代码
COPY . .

# 构建前端（输出到 dist/）
RUN pnpm build

# ========== 阶段 2：运行 ==========
FROM nginx:1.25-alpine

# 复制 Nginx 配置
COPY nginx.conf /etc/nginx/nginx.conf

# 复制前端构建产物
COPY --from=builder /app/dist /usr/share/nginx/html

# 暴露端口
EXPOSE 80

# 启动 Nginx
CMD ["nginx", "-g", "daemon off;"]
```

#### 4.2.2 后端 Dockerfile（多阶段构建）

```dockerfile
# ========== 阶段 1：构建 ==========
FROM node:22-alpine AS builder

# 设置工作目录
WORKDIR /app

# 复制依赖声明文件
COPY package*.json pnpm-lock.yaml ./

# 安装依赖
RUN npm install -g pnpm && pnpm install --frozen-lockfile

# 复制源代码
COPY . .

# 编译 TypeScript
RUN pnpm build

# 生成 Prisma Client
RUN npx prisma generate

# ========== 阶段 2：运行 ==========
FROM node:22-alpine

# 设置工作目录
WORKDIR /app

# 复制依赖声明文件
COPY package*.json pnpm-lock.yaml ./

# 安装生产依赖（不安装 devDependencies）
RUN npm install -g pnpm && pnpm install --frozen-lockfile --prod

# 复制构建产物
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/prisma ./prisma

# 生成 Prisma Client
RUN npx prisma generate

# 非 root 用户运行（安全加固）
RUN addgroup -S nodejs && adduser -S nodejs -G nodejs
USER nodejs

# 暴露端口
EXPOSE 3000

# 健康检查
HEALTHCHECK --interval=30s --timeout=10s --start-period=40s --retries=3 \
  CMD node -e "require('http').get('http://localhost:3000/health', (r) => {process.exit(r.statusCode === 200 ? 0 : 1)})"

# 启动应用
CMD ["node", "dist/server.js"]
```

### 4.3 Docker Compose 配置（本地开发）

```yaml
# docker-compose.yml（本地开发）
version: '3.8'

services:
  # 前端服务
  web:
    build:
      context: ./apps/web
      dockerfile: Dockerfile.dev # 开发环境 Dockerfile（含热重载）
    ports:
      - '5173:5173'
    volumes:
      - ./apps/web:/app
      - /app/node_modules # 匿名卷，避免覆盖本地 node_modules
    environment:
      - VITE_API_BASE_URL=http://localhost:3000
    depends_on:
      - api

  # 后端服务
  api:
    build:
      context: ./apps/api
      dockerfile: Dockerfile.dev # 开发环境 Dockerfile（含热重载）
    ports:
      - '3000:3000'
    volumes:
      - ./apps/api:/app
      - /app/node_modules
    environment:
      - DATABASE_URL=postgresql://postgres:postgres@postgres:5432/pathwise
      - REDIS_URL=redis://redis:6379
      - NODE_ENV=development
    depends_on:
      - postgres
      - redis

  # PostgreSQL 数据库
  postgres:
    image: postgres:16-alpine
    ports:
      - '5432:5432'
    environment:
      - POSTGRES_USER=postgres
      - POSTGRES_PASSWORD=postgres
      - POSTGRES_DB=pathwise
    volumes:
      - postgres_data:/var/lib/postgresql/data
    healthcheck:
      test: ['CMD-SHELL', 'pg_isready -U postgres']
      interval: 10s
      timeout: 5s
      retries: 5

  # Redis 缓存
  redis:
    image: redis:7-alpine
    ports:
      - '6379:6379'
    volumes:
      - redis_data:/data
    command: redis-server --appendonly yes # 持久化

volumes:
  postgres_data:
  redis_data:
```

## 5. CI/CD 流水线设计

### 5.1 流水线概览

```
┌─────────────────────────────────────────────────────┐
│                  代码提交 (Git Push)                  │
└─────────────────┬───────────────────────────────────┘
                  │
                  ▼
┌─────────────────────────────────────────────────────┐
│                 阶段 1：代码质量检查                   │
│  - Lint（ESLint）                                 │
│  - 类型检查（TypeScript）                          │
│  - 单元测试（Jest）                               │
│  - 代码覆盖率检查（≥80%）                         │
└─────────────────┬───────────────────────────────────┘
                  │ 通过
                  ▼
┌─────────────────────────────────────────────────────┐
│                 阶段 2：构建 Docker 镜像              │
│  - 前端镜像（web:${GIT_SHA}）                    │
│  - 后端镜像（api:${GIT_SHA}）                    │
│  - 推送到镜像仓库（阿里云容器镜像服务）             │
└─────────────────┬───────────────────────────────────┘
                  │ 通过
                  ▼
┌─────────────────────────────────────────────────────┐
│                 阶段 3：自动化测试                    │
│  - 集成测试                                        │
│  - API 测试                                        │
│  - E2E 测试（Playwright）                         │
└─────────────────┬───────────────────────────────────┘
                  │ 通过
                  ▼
┌─────────────────────────────────────────────────────┐
│                 阶段 4：部署到测试环境                 │
│  - 拉取镜像                                        │
│  - 更新 Kubernetes Deployment                      │
│  - 健康检查                                        │
└─────────────────┬───────────────────────────────────┘
                  │ 通过（手动审批）
                  ▼
┌─────────────────────────────────────────────────────┐
│                 阶段 5：部署到预发布环境               │
│  - 蓝绿部署                                        │
│  - 最终验收测试                                    │
└─────────────────┬───────────────────────────────────┘
                  │ 通过（手动审批）
                  ▼
┌─────────────────────────────────────────────────────┐
│                 阶段 6：部署到生产环境                 │
│  - 蓝绿部署                                        │
│  - 流量切换                                        │
│  - 监控告警                                        │
└─────────────────────────────────────────────────────┘
```

### 5.2 CI/CD 工具选型

| 工具                   | 用途         | 推荐                      |
| ---------------------- | ------------ | ------------------------- |
| **GitHub Actions**     | CI/CD 流水线 | ✅ 推荐（与 GitHub 集成） |
| **GitLab CI/CD**       | CI/CD 流水线 | ✅ 备选（自托管）         |
| **Jenkins**            | CI/CD 流水线 | ❌ 不推荐（配置复杂）     |

### 5.3 GitHub Actions 流水线示例

#### 5.3.1 前端 CI/CD（`.github/workflows/deploy-web.yml`）

```yaml
name: Deploy Web

on:
  push:
    branches: [main]
    paths: ['apps/web/**']
  pull_request:
    branches: [main]
    paths: ['apps/web/**']

jobs:
  # 作业 1：代码质量检查
  lint-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: 'pnpm'

      - name: Install pnpm
        run: npm install -g pnpm

      - name: Install dependencies
        run: pnpm install --frozen-lockfile

      - name: Lint
        run: pnpm --filter web lint

      - name: Type check
        run: pnpm --filter web type-check

      - name: Unit tests
        run: pnpm --filter web test -- --coverage

      - name: Upload coverage
        uses: codecov/codecov-action@v4
        with:
          token: ${{ secrets.CODECOV_TOKEN }}
          flags: web

  # 作业 2：构建和部署（仅 main 分支）
  deploy:
    needs: lint-and-test
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: 'pnpm'

      - name: Install pnpm
        run: npm install -g pnpm

      - name: Install dependencies
        run: pnpm install --frozen-lockfile

      - name: Build
        run: pnpm --filter web build

      - name: Deploy to Vercel
        uses: amondnet/vercel-action@v25
        with:
          vercel-token: ${{ secrets.VERCEL_TOKEN }}
          vercel-org-id: ${{ secrets.VERCEL_ORG_ID }}
          vercel-project-id: ${{ secrets.VERCEL_PROJECT_ID }}
          working-directory: apps/web
          vercel-args: '--prod'
```

#### 5.3.2 后端 CI/CD（`.github/workflows/deploy-api.yml`）

```yaml
name: Deploy API

on:
  push:
    branches: [main]
    paths: ['apps/api/**', 'prisma/**']
  pull_request:
    branches: [main]
    paths: ['apps/api/**', 'prisma/**']

jobs:
  # 作业 1：代码质量检查
  lint-and-test:
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:16-alpine
        env:
          POSTGRES_USER: postgres
          POSTGRES_PASSWORD: postgres
          POSTGRES_DB: test_db
        ports:
          - 5432:5432
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5

      redis:
        image: redis:7-alpine
        ports:
          - 6379:6379
        options: >-
          --health-cmd "redis-cli ping"
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5

    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: 'pnpm'

      - name: Install pnpm
        run: npm install -g pnpm

      - name: Install dependencies
        run: pnpm install --frozen-lockfile

      - name: Lint
        run: pnpm --filter api lint

      - name: Type check
        run: pnpm --filter api type-check

      - name: Run database migrations
        run: pnpm --filter api prisma migrate deploy
        env:
          DATABASE_URL: postgresql://postgres:postgres@localhost:5432/test_db

      - name: Unit tests
        run: pnpm --filter api test -- --coverage
        env:
          DATABASE_URL: postgresql://postgres:postgres@localhost:5432/test_db
          REDIS_URL: redis://localhost:6379

      - name: Upload coverage
        uses: codecov/codecov-action@v4
        with:
          token: ${{ secrets.CODECOV_TOKEN }}
          flags: api

  # 作业 2：构建 Docker 镜像
  build-and-push:
    needs: lint-and-test
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Log in to Aliyun Container Registry
        uses: docker/login-action@v3
        with:
          registry: registry.cn-hangzhou.aliyuncs.com
          username: ${{ secrets.ALIYUN_USERNAME }}
          password: ${{ secrets.ALIYUN_PASSWORD }}

      - name: Build and push Docker image
        uses: docker/build-push-action@v5
        with:
          context: ./apps/api
          file: ./apps/api/Dockerfile
          push: true
          tags: |
            registry.cn-hangzhou.aliyuncs.com/pathwise/api:latest
            registry.cn-hangzhou.aliyuncs.com/pathwise/api:${{ github.sha }}

  # 作业 3：部署到生产环境
  deploy:
    needs: build-and-push
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to Aliyun ECS
        uses: appleboy/ssh-action@v1.0.0
        with:
          host: ${{ secrets.ALIYUN_ECS_HOST }}
          username: ${{ secrets.ALIYUN_ECS_USER }}
          key: ${{ secrets.ALIYUN_ECS_SSH_KEY }}
          script: |
            # 拉取最新镜像
            docker pull registry.cn-hangzhou.aliyuncs.com/pathwise/api:${{ github.sha }}

            # 更新 Docker Compose 配置
            cd /opt/pathwise
            sed -i "s|image: .*|image: registry.cn-hangzhou.aliyuncs.com/pathwise/api:${{ github.sha }}|" docker-compose.prod.yml

            # 滚动更新（零停机）
            docker compose -f docker-compose.prod.yml up -d --no-deps --build api

            # 健康检查
            sleep 10
            curl -f http://localhost:3000/health || exit 1

            # 清理旧镜像
            docker image prune -f
```

## 6. 监控与日志

### 6.1 监控指标

| 类型         | 指标            | 告警阈值         |
| ------------ | --------------- | ---------------- |
| **应用性能** | 响应时间（P95） | > 500ms          |
|              | 错误率          | > 1%             |
|              | QPS             | -                |
| **系统资源** | CPU 使用率      | > 80%            |
|              | 内存使用率      | > 90%            |
|              | 磁盘使用率      | > 85%            |
| **数据库**   | 连接数          | > 80% 最大连接数 |
|              | 慢查询（>1s）   | > 10 个/分钟     |
| **缓存**     | 命中率          | < 90%            |
|              | 内存使用率      | > 90%            |
| **LLM API**  | 调用成功率      | < 95%            |
|              | 平均响应时间    | > 5s             |

### 6.2 监控工具选型

| 工具               | 用途                | 推荐                          |

## 7. 灾难恢复

### 7.1 备份策略

| 数据类型   | 备份频率              | 保留时间       | 备份方式              |
| ---------- | --------------------- | -------------- | --------------------- |
| **数据库** | 每日全量 + 每小时增量 | 30 天          | PostgreSQL WAL 归档   |
| **文件**   | 实时同步              | 永久           | 阿里云 OSS 跨区域复制 |
| **配置**   | 每次变更              | 永久           | Git 仓库              |
| **镜像**   | 每次构建              | 最近 10 个版本 | 阿里云容器镜像服务    |

### 7.2 RTO & RPO 目标

| 指标                    | 定义                   | 目标     |
| ----------------------- | ---------------------- | -------- |
| **RTO（恢复时间目标）** | 系统恢复可用的最长时间 | < 1 小时 |
| **RPO（恢复点目标）**   | 数据丢失的最长时间窗口 | < 1 小时 |

### 7.3 灾难恢复演练

**频率**：每季度一次

**演练场景**：

1. 数据库主实例故障 → 切换到从实例
2. 可用区故障 → 切换到异地备份
3. 人为误操作 → 数据恢复
